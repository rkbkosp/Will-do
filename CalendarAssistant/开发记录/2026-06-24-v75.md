## 二、悬浮窗语音输入改造

### 2.1 录音后直接保存普通语音条

原流程中，悬浮窗语音录制结束后会进入 `CONFIRMING` 状态，并显示两个按钮：保存、保存为待办。用户希望去掉这一步，让语音先安全落为普通随口记。

本轮调整：

- `FloatingScheduleService.stopVoiceCapture()` 在录音有效时直接调用保存逻辑。
- 外部语音录制完成回调 `handleExternalVoiceCompleted(...)` 同样直接保存。
- 默认保存为普通语音随口记，`asTodo = false`。
- 旧 `confirmVoiceCapture(...)` 保留为兼容入口，但正常流程不再进入两按钮确认态。
- `VoiceCaptureContent` 的 `CONFIRMING` 分支不再显示两个操作按钮，只作为安全兜底状态。

效果：

- 录音结束后立即保存。
- 语音条自然出现在悬浮窗随口记列表中。
- 不再要求用户在录音结束后立即做“普通/待办”决策。

### 2.2 输入框保留 5 秒语音条

在立即保存后，悬浮窗底部输入框会临时展示刚保存的语音条，持续约 5 秒。

实现要点：

- `FloatingScheduleService` 增加 `recentVoiceMemoId` 状态。
- `quickMemoCenter.createVoiceMemo(...)` 返回 memo id 后，调用 `showRecentVoiceMemo(memoId)`。
- 5 秒后自动清除临时输入框状态，语音条仍保留在随口记列表中。
- 如果用户点击发送到实况通知，则立即清除临时语音条。

底部输入框展示：

- 左侧显示语音波形和转写状态。
- 转写中或失败时右侧发送按钮不可用。
- 转写成功且文本非空后，右侧发送按钮可用。

### 2.3 语音转写发送到实况通知

用户明确希望“用实况通知，把转译文本作为 title 创建实况通知”。本轮没有走普通通知，而是接入既有胶囊链路：

```text
FloatingScheduleService
→ CapsuleCommandApi
→ CapsuleStateManager
→ CapsuleDispatcher
→ 原生实况通知 / Flyme / 小米超级岛
```

新增/调整内容：

- 新增 `CapsuleType.VOICE_TRANSCRIPTION`。
- 新增 `CapsuleCommandApi.showVoiceTranscription(...)` 与清理方法。
- `CapsuleStateManager` 增加语音转写临时胶囊状态。
- `CapsuleMessageComposer` 与 `SystemLiveDisplay` 增加语音转写 display model。
- `NotificationIds` 增加稳定 id：`quickMemoVoiceTranscription(memoId)`。
- `IconUtils`、Flyme 模板、小米超级岛 summary 映射增加语音转写类型支持。

展示规则：

- `primaryText` 使用完整转写文本，满足“转写文本作为 title”。
- `shortText` 使用紧凑文本，供折叠态/短标题使用。
- 小米超级岛不强制使用短标题，优先展示主标题。

### 2.4 成功 Toast 移除

用户反馈 `已保存到随口记`、`已挂到实况通知` Toast 会遮挡悬浮窗输入框。

本轮移除：

- 语音保存成功 Toast。
- 语音转写挂到实况通知成功 Toast。
- 文本随口记保存成功 Toast。

保留：

- 权限、错误、配置未开启、转写未完成等异常提示 Toast。

### 2.5 发送按钮样式统一

语音转写发送到实况通知的按钮改为与普通日程悬浮窗发送按钮一致：

- 使用同一个上箭头图标。
- 不再使用上传云图标。
- 有可发送内容时启用，否则禁用。

## 四、每日提醒与明日预告时间自定义

### 4.1 设置模型与配置登记

新增两个用户可调配置：

```text
notification.user.daily_summary_morning_minute_of_day
notification.user.daily_summary_evening_minute_of_day
```

对应设置字段：

- `MySettings.dailySummaryMorningMinuteOfDay`
- `MySettings.dailySummaryEveningMinuteOfDay`

默认值：

- 今日提醒：06:00
- 明日预告：22:00

分钟值统一通过 `MySettings.normalizeDailySummaryMinuteOfDay(...)` 归一化到 `0..1439`。

### 4.2 调度逻辑接入

`DailySummaryReceiver` 不再硬编码 06:00 / 22:00，而是读取当前设置：

- `TYPE_MORNING` 使用 `dailySummaryMorningMinuteOfDay`。
- `TYPE_EVENING` 使用 `dailySummaryEveningMinuteOfDay`。
- 每次 receiver 执行后用最新设置重新调度下一次闹钟。

设置页改时间后，通过 `SettingsViewModel.updateDailySummaryTimes(...)` 保存完成再触发 `runtimeCenter.scheduleDailySummary()`，避免读取旧设置。

### 4.3 设置页滚轮选择器

用户希望时间选择器样式参考“日程默认持续时间”。本轮新增同风格组件：

- `DailySummaryTimeSettingItem`
- `DailySummaryTimePickerDialog`
- `DailySummaryTimeWheelPicker`

特点：

- 使用 `AlertDialog`。
- 标题样式、圆角、按钮风格与默认持续时间选择器一致。
- 小时、分钟两列滚轮选择。

设置位置：

```text

```

开启“每日提醒”后显示：

- 今日提醒时间。
- 明日预告时间。

## 五、正则识别主链路

### 5.1 识别模式设置

新增识别模式设置：

```text
recognition.user.mode
```

模式定义在 `RecognitionMode`：

- `AI_ONLY`：仅 AI。
- `REGEX_ONLY`：仅正则。
- `REGEX_THEN_AI_ON_EMPTY`：正则优先，失败后 AI。
- `REGEX_THEN_AI_REVIEW`：正则先入库，AI 后台修正。

设置入口位于：

```text
偏好设置 → AI → 识别模式
```

选择器使用滚轮样式，与其它设置页滚轮选择器保持一致。

### 5.2 台账登记

按项目约定先登记再实现，本轮新增/调整了以下台账：

- `FeatureCatalog`：新增“正则日程识别”。
- `ConfigCatalog`：新增识别域与识别模式配置项。
- `PageCatalog`：新增开发者“正则规则”页面。
- `HelperCatalog`：新增正则解析与正则规则偏好存储。
- `PolicyCatalog`：将识别模式策略从规划项转为 active。
- `PipelineCatalog`：新增“正则入库后 AI 复核”流程记录。

### 5.3 路由接入

文本识别统一通过 `RecognitionTextNode` 路由：

- `parseUserText(...)`：悬浮窗手动输入、笔记转日程等单条识别入口。
- `analyzeTextEvents(...)`：随口记语音转写建议等多条文本识别入口。

路由策略：

- `AI_ONLY`：保持原 AI 路径。
- `REGEX_ONLY`：只走正则规则。
- `REGEX_THEN_AI_ON_EMPTY`：正则无结果时同步调用 AI。
- `REGEX_THEN_AI_REVIEW`：正则命中后立即返回并入库，后续由入库成功事件触发 AI 复核。

图片/截图识别仍保持原 OCR/多模态 AI 路径，未在本轮强行切到正则。

## 七、正则规则模型与默认规则

### 7.1 规则模型

`RegexScheduleRule` 字段包括：

- `id`
- `name`
- `enabled`
- `pattern`
- `titleTemplate`
- `tag`
- `dateGroup`
- `timeGroup`
- `titleGroup`
- `locationGroup`
- `locationTemplate`
- `descriptionTemplate`

其中 `locationTemplate` 是本轮后续补充的字段，用来让正则结果也能生成类似 AI 的路线地点：

```text

```

保留两个基础规则：

- 日期时间标题。
- 标题日期时间。

普通 `general` 文本形态过于开放，本轮没有继续扩大多行正则范围，避免正则先入库阶段产生过多误报。复杂普通通知仍建议交给 AI 修正兜底。

### 7.3 码类规则

新增多行 detail 规则，并保留原单行 code 规则作为兜底：

- 取件通知多行。
- 取餐通知多行。
- 取票通知多行。
- 寄件通知多行。
- 取件码。
- 取餐码。
- 取票码。
- 寄件码。

多行规则优先提取：

- code。
- brand。
- location。
- time。

并统一通过 `RecognitionRuleCatalog.formatDescription(...)` 生成描述头：

```text
【取件】取件码|品牌|位置
【取餐】取餐码|品牌|位置
【取票】取票码|品牌|位置
【寄件】寄件码|品牌|地点
```

### 7.4 交通类规则

新增三类默认规则：

- 列车订单。
- 航班订单。
- 打车行程。

这些规则主要适配 OCR 多行结构，因为真实样本中车次、时间、日期、出发地、到达地、检票口、座位号经常分散在不同行。

列车输出目标：

```text

description: 【列车】G1732|A12|03车05A二等座

```

航班输出目标：

```text

location: 武汉(WUH) -> 北京首都(PEK)
description: 【航班】MU2456|23|12A

```

打车输出目标：

```text

location: 湖北大学南门 -> 光谷广场地铁站A口
```

### 7.5 匹配排序

为避免交通或码类文本先被普通日程规则命中成 `general`，`RegexScheduleRecognizer` 增加排序策略：

- 具体 tag 优先于 `general`。
- `_detail` / `_order` 规则优先于单行兜底规则。
- 同优先级下保留规则原始顺序。

这样旧设备即使通过默认规则合并把新增规则追加到列表末尾，也能优先命中更具体的 detail/order 规则。

## 八、正则测试崩溃修复

开发者正则规则页点击“测试”曾出现崩溃。真机日志显示：

```text
PatternSyntaxException: Syntax error in regexp pattern
pattern: \{([A-Za-z][A-Za-z0-9_]*)}
```

原因是模板占位符正则只转义了左花括号，没有转义右花括号，Android ICU 正则解析失败。

修复：

```kotlin
Regex("\\{([A-Za-z][A-Za-z0-9_]*)\\}")
```

同时 `RegexRuleEditorPage.runTest()` 外层增加 `runCatching`，坏规则或异常模板不会再导致页面崩溃，只显示测试失败信息。

## 九、正则先入库后 AI 后台修正

### 9.1 设计原则

用户希望“正则先入库，AI 后台分析修正”，但也明确担心额度消耗。本轮将该行为作为显式识别模式，而不是默认静默开启。

模式名称：

```text
正则先入库，AI 后台修正
```

说明文案明确提示：

```text
会消耗 AI 额度
```

### 9.2 事件链路

识别层仍只产生候选结果，不直接修改数据库。AI 修正放在入库成功之后执行：

```text
RecognitionCenter
→ DomainEventBus.RECOGNITION_COMPLETED(rawText + regex draft)
→ ContentIngestCenter
→ ImportCenter / ScheduleCenter.addEvent
→ INGEST_SUCCEEDED(createdEventIds)
→ RegexAiReviewCoordinator
→ RecognitionTextNode.analyzeTextEventsByAi(...)
→ ScheduleCenter.updateEvent(...)
```

### 9.3 安全限制

后台修正只在以下条件同时满足时执行：

- 当前模式是 `REGEX_THEN_AI_REVIEW`。
- 原始识别文本非空。
- 正则候选结果只有一条。
- 入库成功且新建事件只有一条。
- AI 返回结果只有一条。
- AI 返回前用户没有修改该事件。
- AI 结果与当前事件确有差异。

以下情况直接跳过：

- 正则未命中。
- 入库被去重。
- AI 返回空、失败或多条。
- 用户在 AI 返回前已经编辑过事件。
- 当前事件已被删除。

后台复核强制调用 AI 路径：

```text
RecognitionTextNode.analyzeTextEventsByAi(...)
```

不会递归回当前识别模式。
