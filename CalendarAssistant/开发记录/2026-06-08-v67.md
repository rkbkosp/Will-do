## 二、通知生命周期统一收口

### 2.1 问题背景

用户在列车事件的实况通知中点击“已检票”后，理论上通知应该从“检票口/待检票”切换到“车厢/座位号”。

实际表现是：点击“已检票”后，列车实况通知短暂消失；过一段时间天气或胶囊兜底刷新触发后，车厢和座位号又重新出现。

这说明不是用户手动划掉通知，而是某条旧通知链路先取消了胶囊，后续胶囊状态重算又把它恢复。

### 2.2 根因

旧链路中存在多个通知注册/取消入口：

- `StoreRootNode` 写库后直接调用 `ReminderStoreNode`。
- `ReminderStoreNode.rebuildForEvent()` 将 `state != STATE_PENDING` 视为不活跃事件。
- 不活跃事件清理会同时取消普通提醒和 `NotificationIds.liveCapsule(eventId)`。
- `CapsuleStateManager` 自己的活跃判断又没有把 `STATE_CHECKED_IN` 当成完成态。

因此列车进入 `STATE_CHECKED_IN` 后，旧提醒节点误判为不活跃，取消了实况胶囊；之后胶囊状态重算又认为该列车仍然有效，于是重新发出通知。

### 2.3 新链路方案

本轮按 4 月底 center/port/node 重构口径，把日程通知生命周期收口到 `ReminderCenter`：

```text
CalendarCenter 写操作 -> schedule.changed -> ReminderCenter -> Alarm/Notification/Capsule refresh
```

核心变化：

1. `CalendarCenter` 作为写库外层端口，写操作完成后发出 `DomainEventType.SCHEDULE_CHANGED`。
2. `ReminderCenter` 订阅 `schedule.changed`，统一负责日程提醒、胶囊闹钟和可见通知的重算。
3. `StoreRootNode` 退回纯数据层职责，不再在写库后直接注册或取消通知。
4. `ScheduleCenter` 移除旧的通知窗口刷新 debounce，避免和 `ReminderCenter` 形成双轨。
5. `BootReceiver` 和 `ReminderReconcileReceiver` 统一调用 `ReminderCenter.reconcileAll()` / `reconcileAllNow()`。

### 2.4 重建策略

新链路更偏“暴力但可靠”：

- App 启动、开机、应用更新和兜底定时器会基于数据库重新计算未过期日程。
- 日程变更事件触发后，`ReminderCenter` 先显式取消删除/归档 ID，再全量重建当前未过期日程。
- 重算时直接读取最新数据库快照，避免依赖 `ScheduleCenter` 内存列表刷新时序。
- 标准提醒和实况胶囊不再共用 `state != PENDING` 这样的粗粒度失效判断。

### 2.5 防御性修复

在旧底层工具仍被保留为内部实现的前提下，补充了防御性语义：

- `ReminderStoreNode` 对 `STATE_CHECKED_IN && isTransit` 不再取消可见实况胶囊。
- `NotificationScheduler.cancelReminders()` 拆出只取消闹钟的 `cancelScheduledAlarms()` 和取消可见通知的 `cancelVisibleNotifications()`。
- `ReminderCenter` 重排时不再调用会误取消可见胶囊的旧“一把梭”取消逻辑。

这样即使未来还有旧入口残留，也不容易再次把已检票列车的车厢/座位通知误清理。

## 三、原生实况通知模板中心

### 3.1 问题背景

ColorOS/OPlus 原生实况通知存在严格行数限制。此前曾尝试通过原生字段、探针和 `BigTextStyle` 拓展显示，但实机表现不稳定。

同时，OneUI、Pixel/AOSP、LineageOS/PixelOS 等系统通常可以展示更多正文内容。

因此不能把所有系统都强行压成两行，也不能让 ColorOS 继续展示完整长文本。

### 3.2 新增 NotificationTemplateCenter

新增 `NotificationTemplateCenter`，统一决定原生实况通知使用哪种模板：

```kotlin
enum class NotificationTemplateMode {
    FULL_MULTILINE,
    COMPACT_TWO_LINE
}
```

模板中心明确分离：

- `primaryText`：通知顶部标题。
- `secondaryText`：正文第 1 行。
- `tertiaryText`：正文第 2 行。
- `expandedText`：展开态完整正文。

这样避免出现 OPPO 实况卡片中“顶部 title + 正文第一行又是 title”的重复显示问题。

### 3.3 系统能力识别

`OsUtils` 新增系统识别与原生多行能力判断：

- `isColorOsLike()`：识别 OPPO、OnePlus、realme、OPlus 和 ColorOS 系统属性。
- `isOneUi()`：识别 Samsung/OneUI。
- `isPixelLike()`：识别 Google Pixel/AOSP 风格。
- `isLineageLike()`：识别 LineageOS/PixelOS。
- `supportsNativeMultilineLiveNotification()`：输出原生实况通知是否应默认走完整多行模板。

默认策略：

- ColorOS/OPlus：`COMPACT_TWO_LINE`。
- OneUI/Pixel/Lineage：`FULL_MULTILINE`。
- 其他系统：默认完整。

同时预留 `COLOR_OS_MULTILINE_ALLOWLIST`，后续如果确认某些 ColorOS 构建支持完整展示，可以单独放行。

## 四、开发者模板开关

### 4.1 问题背景

仅靠 ROM 自动识别不够灵活：

- 某些 ColorOS 版本可能重新支持完整展示。
- 某些非 ColorOS 系统也可能存在两行限制。
- 开发阶段需要快速对比完整模板和精简模板效果。

因此本轮新增开发者选项开关，让模板中心不再只是“自动策略”。

### 4.2 新增设置字段

`MySettings` 新增：

```kotlin
liveNotificationTemplateMode: String = LiveNotificationTemplateMode.AUTO
```

可选值：

```text
AUTO
FULL
COMPACT
```

设置链路接入：

- `SettingsTransformApi`
- `LocalSettingsTransformApi`
- `SettingsViewModel.updatePreference()`

保存时通过 `LiveNotificationTemplateMode.normalize()` 兜底，非法值统一回到 `AUTO`。

### 4.3 开发者选项入口

`LaboratoryPage` 的开发者选项中新增“原生实况通知模板”：

- 自动
- 完整
- 精简

切换后会：

1. 保存设置。
2. 调用 `capsuleCenter.forceRefresh()`。
3. Toast 提示当前模式。

### 4.4 生效范围

该开关只影响原生实况通知模板，不改变 Flyme、HyperOS/MIUI 岛和其他自定义通道的 Provider 策略。

天气临时胶囊现在会在状态中保留原始 `WeatherAlertData` / `WeatherRiskAlert`，因此设置变化后可以用新模板重渲染当前天气胶囊。

## 五、日程模板策略

### 5.1 顶部标题与正文分离

普通日程的顶部标题仍使用日程标题。

正文不再重复标题，而是从以下字段中生成：

```text
location
description
```

### 5.2 完整模板

`FULL_MULTILINE` 下，普通日程正文按固定顺序展示：

```text
time
location
description
```

适用于 OneUI、Pixel、Lineage 等支持多行展示的系统。

### 5.3 精简模板

`COMPACT_TWO_LINE` 下，普通日程正文使用关键信息优先策略：

1. 第 1 行：`description > location > time`。
2. 第 2 行：优先补 `location`，然后补 `description`，最后补 `time`。
3. 不重复顶部 `title`。

这符合 ColorOS 两行限制下“用户更关心备注/地点，实在没有再看时间”的讨论结论。

### 5.4 特殊规则保持独立

列车、航班、取件、取餐、取票、寄件等特殊类型仍使用现有解析字段：

- 列车：检票口/座位 + 车次。
- 航班：登机口/座位。
- 取件/取餐：取件码 + 平台/地点。

这些类型不直接展示微格式 `description`，避免把内部识别格式暴露给用户。

## 六、天气模板策略

### 6.1 官方天气预警

官方预警不使用生活化建议，而是展示“权威标题 + 关键事实”：

- 顶部标题：`地点 · 红色暴雨预警`、`厦门 · 黄色大风预警`。
- 完整模板：持续时间、发布单位、完整 `description`、`instruction`。
- 精简模板：关键事实 + 时间兜底。

关键事实抽取采用“结构字段优先 + 规则抽取 + 正则兜底”：

- 解除类：优先展示解除原因，如“强雷雨已移出，预警解除”。
- 降雨类：提取 `1-2小时内`、`20-40毫米降水` 等短事实。
- 大风类：提取 `未来12小时阵风7-8级` 等风力信息。
- 地灾类：提取“地质灾害风险较高”。
- 高温类：提取最高气温等信息。
- 抽不到时用 `持续至xx:xx`、`xx:xx发布` 等时间字段兜底。

### 6.2 本地天气风险

本地风险精简模板采用两句生活化文案：

```text
预计约2小时后降雨，降水概率80%
建议出门带伞，注意路滑和延误
```

不同风险类型使用不同短建议：

- 降雨：出门带伞，注意路滑和延误。
- 雷电/强对流：减少户外，远离树下高处。
- 高温：多喝水，少晒太阳。
- 降雪/结冰：慢点走，预留出行时间。
- 大风/台风：收好窗边和阳台物品。
- 雾霾/沙尘：戴好口罩再出门。

完整模板继续使用现有 `risk.message`，不丢原始说明。

## 七、天气风险图标修复

### 7.1 问题背景

实况通知中“大风”天气风险显示成了下雨图标。

排查发现：

- 本地大风风险的正文里可能包含 `降水概率`、`预计降水` 等字段。
- 旧 `WeatherAlertIconMapper.iconRes()` 会把标题、正文、展开文本全部拼接后平权匹配。
- 匹配顺序中雨类在风类之前。

因此“大风 + 降水概率”会先命中雨图标。

### 7.2 修复方式

`WeatherAlertIconMapper.riskIconRes()` 改为风险专用策略：

1. 优先按 `risk.title` 判断主类型。
2. 再按 `risk.weatherText` 判断。
3. 最后才用 `risk.message` 兜底。

`IconUtils` 中 `weather_risk` 胶囊也改为走风险专用图标入口，不再走普通全文平权匹配。

这样“大风”即使正文包含降水概率，也会优先显示风图标。
