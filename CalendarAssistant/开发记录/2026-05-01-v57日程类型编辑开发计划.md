# 2026-05-01 开发计划 (v57)：日程类型编辑与新建弹窗精简

## 一、背景

当前新建/编辑日程弹窗 `AddEventDialog` 以通用日程为主，虽然已有 `EventTags.TRAIN`、`EventTags.PICKUP`、`EventTags.FOOD` 等事件类型，并且展示、胶囊、通知链路已经能按类型渲染，但用户手动创建时仍缺少直接选择“列车 / 取件 / 取餐”等类型的入口。

同时，弹窗里的“添加提醒”入口使用价值较低，并且会增加后续类型化表单的复杂度。后续计划先从 UI 层移除手动提醒入口，保留底层提醒字段和全局提醒策略，避免影响现有通知、胶囊、同步和重复日程链路。

---

## 二、目标

1. 在新建日程弹窗中支持直接选择事件类型，例如：日程、列车、取件、取餐。
2. 用户点击类型区域后，弹出或展开可横向滑动的类型选项。
3. 选择非“日程”类型后，弹窗下半部分切换为对应类型的专用输入区。
4. 专用输入区提交时将结构化字段拼接进 `description`，并写入对应 `tag`。
5. 移除“添加提醒”手动入口，降低弹窗复杂度。
6. 第一版不改数据库表结构，不新增事件事实源。

---

## 三、设计原则

- 继续使用现有 `Event` 模型，不新增列车、取件等独立实体。
- 使用 `tag + description 微格式` 表达类型化内容，复用现有 `RuleMatchingEngine` 和 `EventPresentationInternals`。
- `tag` 与 `description` 头部保持一致，避免保存后被标签推断逻辑覆盖。
- UI 做类型化输入，但落库仍走现有 `EventPatch -> ScheduleCenter -> CalendarCenter -> StoreRootNode` 链路。
- 第一版只做手动创建/编辑体验，不影响 AI 识别、短信识别、系统日历同步主链。

---

## 四、弹窗交互草案

### 1) 类型选择入口

当前顶部类型区域显示为类似：

```text
类型: [日程]
```

计划改为可点击区域：

```text
类型: [日程 v]
```

点击后展开横向类型选择条：

```text
[日程] [列车] [取件] [取餐] [航班] [取票] [寄件] ...
```

第一版建议先开放：

- 日程
- 列车
- 取件
- 取餐

后续再扩展航班、取票、寄件、用车等类型。

### 2) 表单切换动效

选择类型后，弹窗下半部分使用 `AnimatedContent` 或类似方案做淡入/滑入/尺寸变化。

暂不建议第一版做 3D 翻转，原因：

- Dialog 高度变化时更容易抖动。
- 输入法、滚动容器和焦点处理更复杂。
- 淡入/滑入已经足够表达“表单形态切换”。

## 五、各类型字段草案

### 1) 日程

保留通用日程字段：

- 标题
- 开始时间
- 结束时间
- 重复
- 地点
- 备注

移除：

- 添加提醒
- 已选提醒 chips

提交时：

- `tag = EventTags.GENERAL`
- `description = 用户备注`
- 自定义提醒字段保持关闭值。

### 2) 列车

保留字段：

- 标题
- 开始时间
- 结束时间
- 地点

隐藏字段：

- 重复
- 添加提醒
- 备注

新增字段：

- 车次
- 检票口
- 车厢
- 座位号

提交时拼接为：

```text
【列车】车次|检票口|车厢 座位号
```

示例：

```text
【列车】G123|A8|03车 05A
```

写入：

- `tag = EventTags.TRAIN`
- `location = 用户填写地点`
- `description = 上述微格式`
- `rrule = ""`

### 3) 取件

建议字段：

- 标题
- 截止/提醒时间
- 取件码
- 平台/来源
- 取件地点

提交时拼接为：

```text
【取件】取件码|平台|取件地点
```

示例：

```text
【取件】123456|菜鸟|北门驿站
```

写入：

- `tag = EventTags.PICKUP`
- `location = 取件地点`
- `description = 上述微格式`
- `rrule = ""`

### 4) 取餐

建议字段：

- 标题
- 截止/提醒时间
- 取餐码
- 平台/商家
- 取餐地点

提交时拼接为：

```text
【取餐】取餐码|平台|取餐地点
```

示例：

```text
【取餐】88|美团|前台
```

写入：

- `tag = EventTags.FOOD`
- `location = 取餐地点`
- `description = 上述微格式`
- `rrule = ""`

## 六、关于“添加提醒”的处理

计划先移除 `AddEventDialog` 中的手动提醒 UI：

- 删除“添加提醒”按钮。
- 删除已选提醒 chips 展示。
- 删除 `WheelReminderPickerDialog` 在新建/编辑日程弹窗中的入口。
- 提交 `EventPatch` 时不再写入自定义提醒。

暂不建议删除底层提醒字段或 `ReminderPolicy`，因为这些仍被以下链路使用：

- 全局提前提醒。
- 开始时提醒兜底。
- 胶囊通知时序。
- 重复实例提醒注册。
- 系统日历同步兼容。

注意：如果编辑旧事件并保存，旧事件已有的自定义提醒可能会被清空。这是移除手动提醒入口后的自然结果，后续实现时需要明确是否接受。

---

## 七、编辑已有特殊事件

编辑已有列车、取件、取餐事件时，需要反向解析 `description` 微格式并填回专用输入框。

示例：

```text
【列车】G123|A8|03车 05A
```

应解析为：

- 车次：G123
- 检票口：A8
- 车厢/座位：03车 05A

如果解析失败：

- 保留原始 `description`。
- 可以回退到通用日程表单，避免用户保存时丢内容。

---

## 八、实现顺序建议

1. 移除 `AddEventDialog` 手动提醒 UI 和提交入口。
2. 定义弹窗内部的类型描述，例如 `QuickEventType`。
3. 增加类型选择横向列表。
4. 增加通用日程、列车、取件、取餐的表单状态。
5. 提交时按类型构造 `EventPatch`。
6. 增加已有特殊事件的反向解析。
7. 编译验证并手动测试新增/编辑/删除/胶囊展示链路。

---

## 九、待讨论问题

1. “车号”文案是否改为“车厢”，并与“座位号”分开输入。
2. 非日程类型是否一律禁止重复。
3. 编辑旧事件时清空自定义提醒是否可以接受。
4. 取件/取餐的时间字段应该叫“提醒时间”“截止时间”还是继续使用“开始/结束”。
5. 是否需要在系统日历同步时隐藏微格式，或接受微格式出现在系统日历备注中。
