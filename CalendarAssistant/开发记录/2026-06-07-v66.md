## 二、天气通知位置稳定兜底

### 2.1 问题背景

出行场景下，自动定位会随着用户经过不同城市、区县或定位点发生变化。

旧逻辑中，`WeatherRepository.forceRefresh()` 每次刷新天气后，只要命中天气预警或天气风险提醒，就会直接调用 `WeatherNotifier.notifyAlerts()`。

同时，天气通知去重 key 中包含 `locationName`。因此只要位置名变化，就可能绕过旧去重策略，导致移动中每换一个地方就发送一次天气通知。

### 2.2 位置稳定门控

新增 `WeatherLocationStabilityGate`，用于在通知发送前判断当前位置是否稳定。

核心规则：

- 手动定位直接放行，不延迟通知。
- 自动定位默认要求同一位置连续命中 2 次才允许发送通知。
- 位置签名优先使用 `locationId / adm1 / adm2 / name`。
- 如果缺少地名信息，则使用经纬度保留 2 位小数作为兜底签名。
- 天气数据刷新和缓存不受影响，只延迟通知发送。

这样可以减少路过城市、临时定位漂移带来的通知噪声，同时保留天气页面和小组件的正常刷新。

### 2.3 开发者阈值设置

该能力定位为兜底参数，不适合放在普通天气设置页，因此本轮放入开发者选项。

新增设置字段：

```kotlin
weatherLocationStabilityRequiredHits: Int = 2
```

开发者页提供 1/2/3 次选择：

- `1 次`：等价于关闭兜底，自动定位首次命中即通知。
- `2 次`：默认值，兼顾实时性和降噪。
- `3 次`：更严格，适合移动频繁且通知噪声明显的场景。

设置链路通过 `SettingsTransformApi`、`LocalSettingsTransformApi` 和 `SettingsViewModel.updatePreference()` 接入，并在保存时限制到 `1..3`。

## 三、开发者诊断日志增强

### 3.1 问题背景

开发者选项中的日志导出原本只合并本地文件日志：

- APP 日志
- CRASH 日志
- AI_ENGINE 日志

这类日志适合看崩溃和识别流程，但无法覆盖大量运行时 `Log.d/e`、系统服务回调、通知和窗口相关日志。

因此在排查真实设备问题时，现有导出内容不够完整。

### 3.2 新增 LOGCAT 区块

`DiagnosticLogCenter` 现在导出四段内容：

```text
APP
CRASH
AI_ENGINE
LOGCAT
```

`LOGCAT` 区块会尝试导出运行时 logcat，并优先按应用 UID 过滤。

导出策略：

1. 如果有 Shizuku 或 Root 权限，优先走特权进程执行 `logcat`。
2. 首选 `--uid=<app uid>`，尽量获取同一应用 UID 下的完整运行时日志。
3. 普通权限下尝试本地 `logcat --uid=<app uid>`。
4. 如果 UID 过滤不可用，则回退到 `--pid=<current pid>`。
5. 最后再按包名文本和当前 PID 做兜底过滤。

导出命令使用 `-d -v threadtime`，并在用户选择最近 N 分钟时附加 `-T` 时间范围。

### 3.3 权限限制说明

Android 普通应用无法稳定读取全系统 logcat。

因此本轮实现的目标不是绕过系统限制，而是在能力范围内做到：

- 有 Shizuku/Root 时尽量完整。
- 无特权时至少导出本应用可见日志。
- 导出文件内明确标注 logcat 来源、命令和 fallback 说明。

开发者页文案同步调整为“诊断日志”，并提示日志可能包含识别文本、Prompt、模型响应、接口返回和运行时 logcat。

## 四、新建日程弹窗键盘避让修复

### 4.1 问题背景

加入预测性返回手势后，新建日程 Dialog 使用 edge-to-edge 布局。

旧布局只把 `WindowInsets.navigationBars` 传给 `PredictiveBottomDialogHost`，没有处理 `WindowInsets.ime`。

当用户点击备注输入框时，备注区域位于弹窗底部，软键盘弹出后卡片仍按原位置居中，导致备注输入框被键盘挡住。

### 4.2 修复方式

`AddEventDialog` 的弹窗容器 padding 改为：

```kotlin
WindowInsets.ime.union(WindowInsets.navigationBars).asPaddingValues()
```

这样 Dialog 在键盘出现时会同时避让 IME 和导航栏。

修复点只作用于弹窗外层容器，不改变日程字段布局和预测性返回动画。

## 五、悬浮窗编辑保存与长输入光标修复

### 5.1 编辑保存后显示旧内容

悬浮窗日程卡片编辑时，保存链路会调用 `ScheduleCenter.updateSingleFromPatch()`。

数据层本身会更新，但卡片内部有本地草稿状态和展示模型缓存。

旧代码的刷新 key 主要关注 `stableKey / startTS / endTS / state`，没有覆盖 `title / location / description` 等字段。

因此保存后容易出现“数据已经更新，但卡片仍显示旧标题、旧地点或旧备注”的体感问题。

本轮补齐刷新依赖：

- 草稿同步 `LaunchedEffect` 加入 `title / location / description / tag / color`。
- 展示模型 `remember` 加入 `title / location / description`。

这样编辑保存后，卡片和详情区都会根据最新字段重新计算。

### 5.2 长输入框光标不在末尾

悬浮窗内联编辑使用的 `CompactTextField` 原本是 `String` 版 `BasicTextField`。

当外部值初始化为长文本时，组件无法明确管理 selection，容易出现光标没有稳定移动到最后一个字符之后的问题。

本轮把 `CompactTextField` 改为内部维护 `TextFieldValue`：

- 初始化时 selection 放到 `TextRange(value.length)`。
- 外部 `value` 变化且和内部文本不一致时，重新设置文本并把光标放到末尾。
- 用户编辑过程中保留 Compose 自身 selection，不强行覆盖用户在中间编辑的位置。

## 六、主页 UI 自动时间刷新

### 6.1 问题背景

用户反馈应用在后台停留期间，如果日程过期或跨过 24 点，回到主页后 UI 仍可能保持旧状态。

典型表现：

- 已过期日程没有及时变灰、划线或移动排序。
- 跨天后主页仍停留在旧的“今天”。
- 需要手动切换页面，触发重组后才恢复正常。

排查后发现项目里已有“最近日程过期时触发 `_timeTrigger`”的定时器，但存在两个缺口：

1. 定时器没有覆盖跨天时间点。
2. `SwipeableEventItem` 自己把 `isExpired` 用 `remember(item.endTS)` 缓存，时间触发后不一定重新计算。

### 6.2 MainUiState 新增时间信号

`MainUiState` 新增：

```kotlin
today: LocalDate
timeRefreshToken: Long
```

作用：

- `today` 作为 UI 层统一的“今天”来源，避免多个页面各自调用 `LocalDate.now()` 后状态不同步。
- `timeRefreshToken` 作为时间变化指令，传递给列表、排序和卡片过期样式。

### 6.3 合并过期与跨天触发

`MainViewModel` 的时间定时器现在同时等待三类触发点：

- 最近一个未过期日程的结束时间。
- 下一个 00:00。
- 60 秒兜底检查。

任一命中后调用 `publishTimeRefresh()`。

跨天时额外处理：

- 如果用户当前选中日期仍是旧的今天，则自动推进到新的今天。
- 如果用户手动选了其他日期，则不强行跳回今天。

这样既解决跨天刷新，又避免打断用户查看历史或未来日期。

### 6.4 列表和卡片接入

`SwipeableEventItem` 新增 `timeRefreshToken` 参数，过期判断改为依赖：

```kotlin
remember(item.endTS, timeRefreshToken)
```

接入页面包括：

- 首页今日/明日列表。
- 全部日程列表。
- 归档列表。

同时 `HomePage`、`AllEventsPage`、`ArchivesPage` 中涉及“今天”和年份的判断改为使用 `uiState.today`。
