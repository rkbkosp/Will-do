# 2026-05-17 开发记录 (v59)：侧边栏弹性与 OPPO 导入同步修复

## 一、本次目标

当前分支：`fix/notifications-source-images`。

本次开发覆盖两个用户反馈问题：

1. 设置侧边栏顶部贴边、只有内容被底部遮挡时才能上下滚动，正常高度下缺少拖拽反馈，交互显得生硬。
2. 从小米设备导出的日程备份导入 OPPO 后，日程没有稳定同步到 OPPO 系统日历；过一段时间系统同步后，导入日程会从 App 内消失。

## 二、侧边栏顶部间距与弹性滑动

### 1) 问题背景

`SettingsSidebar` 原先仅使用 `verticalScroll(scrollState)`。当侧边栏内容高度超过可视区域时，底部内容被遮挡才会出现滚动；当内容刚好能放下时，上下拖动没有任何反馈。

这导致侧边栏在多数状态下看起来像静态面板，和移动端常见的弹性回弹手感不一致。

### 2) 实现方案

修改文件：

- `app/src/main/java/com/antgskds/calendarassistant/ui/components/SettingsSidebar.kt`

主要调整：

- 在 `statusBarsPadding()` 后增加顶部 `16.dp` padding。
- 保留原有 `verticalScroll(scrollState)`，内容真实超出时仍走标准滚动。
- 外层增加 `NestedScrollConnection`，监听未被滚动容器消费的纵向拖拽。
- 使用 `rubberBandOffset` 对内容做短距离 `offset` 偏移。
- 手指释放或 fling 结束后，用 `Animatable + spring` 回弹到 0。

### 3) 交互结果

- 内容可滚动时：保持正常滚动。
- 内容不可滚动时：上下拖动也会有跟手偏移和回弹。
- 滚到顶部或底部继续拖动时：出现边界弹性反馈。
- 横向侧边栏开关手势未改动。

## 三、OPPO 导入同步问题定位

### 1) 用户反馈

用户从小米设备切换到 OPPO 后，导入日程备份：

- 导入后 App 内短时间能看到日程。
- OPPO 系统日历没有稳定出现这些日程。
- 过一段时间系统同步后，导入的日程从 App 内消失。

### 2) 备份文件检查

用户提供的备份文件：

```text
C:\Users\ASUS\Downloads\calendar_events_20260517_120356.json
```

统计结果：

```text
version = 3
events = 170
archived = 0
total = 170
importId = { caldav: 169, blank: 1 }
sourceTop = Caldav-2: 78, Caldav-1: 57, Caldav-4: 34, blank: 1
caldavSourceCount = 169
```

结论：备份中绝大多数事件带有旧设备的 `Caldav-*` 远端绑定。

### 3) 根因

手动导入流程原先直接写 Room：

- `LegacyDataMigrationCoordinator.importEventsData()`
- `mergeCandidatesIntoCurrentStore()`
- `dao.insertOrUpdate()`

该流程不会走 `StoreRootNode.createEvent(..., syncToSystem = true)` 的系统日历推送链路。

同时，导入数据中的 `source/importId` 仍指向旧设备的系统日历事件，例如：

```text
source = Caldav-1
importId = Caldav-1-34
```

在 OPPO 上这些远端 id 不存在。后续系统日历拉取时，`SystemCalendarSyncManager.fetchCalDAVCalendarEvents()` 会按当前系统日历结果清理同源但本次没有拉到的本地事件，导致导入日程被当作“远端已删除”并从本地删除。

## 四、导入时旧 CalDAV 绑定清洗

修改文件：

- `app/src/main/java/com/antgskds/calendarassistant/core/migration/LegacyDataMigrationCoordinator.kt`

### 1) 新增导入候选包装

新增 `ImportCandidate`，记录：

- 清洗后的 `event`
- 是否从旧系统日历绑定中剥离
- 原始 `importId`
- 原始 `source`

这样后续合并时既能使用清洗后的事件，又能判断是否需要解绑已有同源旧记录。

### 2) 手动导入专用清洗

新增逻辑：

- 如果 `event.source` 以 `Caldav` 开头，或 `event.importId` 以 `Caldav` 开头：
  - 清空 `importId`
  - 设置 `source = SOURCE_SIMPLE_CALENDAR`
  - 标记 `detachedSystemBinding = true`
- 如果没有系统绑定，只补齐空 `source` 为本地来源。

这一步只应用于手动导入，避免旧设备系统日历 id 直接进入当前设备事实源。

### 3) 跨设备备份的去重策略

当检测到导入内容中存在 `Caldav-*` 绑定时，导入池去重优先使用事件签名：

- 重复事件：标题、地点、标签、规则、开始时间、持续时间。
- 单次事件：标题、地点、标签、开始时间、结束时间、备注。

这样可以避免同一真实日程因为旧设备的不同远端 id 被当作不同事件重复导入。

## 五、导入后补推送到当前系统日历

修改文件：

- `app/src/main/java/com/antgskds/calendarassistant/core/migration/LegacyDataMigrationCoordinator.kt`

### 1) 补推送条件

导入合并完成后，收集需要写入系统日历的事件：

- 未归档。
- 非便签标签。
- `importId` 为空。
- `parentId == 0L`，避免直接推送重复实例子事件。

执行推送前还需要满足：

- `CalendarConfig.caldavSync == true`
- 已授予 `READ_CALENDAR` 和 `WRITE_CALENDAR`

### 2) 推送方式

新增 `pushImportedEventsToSystemCalendar()`：

- 使用 `SystemCalendarSyncManager.insertCalDAVEvent()` 写入当前设备选中的系统日历。
- 推送前确保事件使用空 `importId` 和本地 `source`。
- 推送成功后回写新的 `importId/source` 到原本导入的本地事件 id。
- 如果插入未返回有效系统绑定，只记录警告，不错误删除本地事件。

这样导入后的事件会重新绑定到 OPPO 当前 Calendar Provider，而不是沿用小米设备的旧远端 id。

### 3) 日志

导入完成后记录汇总日志：

```text
Manual event import result: candidates=..., inserted=..., updated=..., skipped=..., queuedForSystem=..., pushedToSystem=...
```

如果同步关闭或日历权限缺失，也会记录跳过原因，方便后续排查。

## 六、同步拉取删除保护

修改文件：

- `app/src/main/java/com/antgskds/calendarassistant/calendar/sync/SystemCalendarSyncManager.kt`

### 1) 原有风险

`fetchCalDAVCalendarEvents()` 拉取系统日历后，会把本地同 `source` 但本次未拉到的事件删除。该策略对正常双向同步成立，但对跨设备导入的旧绑定风险较高。

### 2) 新增保护

新增 `shouldDeleteMissingSystemEvent()`，只有满足以下条件才允许删除：

- `event.source == expectedSource`
- `event.importId` 以当前 `expectedSource-` 开头
- 能解析出有效远端事件 id
- `parentId == 0L`

未满足条件的缺失绑定不直接删除，并输出保护日志：

```text
fetchCalDAVCalendarEvents protected missing local bindings source=... count=...
```

这可以减少异常绑定、半清洗导入数据、重复实例子事件被同步清理链误删的概率。
