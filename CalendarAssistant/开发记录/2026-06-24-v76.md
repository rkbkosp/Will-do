## 三、背景图片导入与取色

### 3.1 私有图片存储

新增文件：

```text
app/src/main/java/com/antgskds/calendarassistant/feature/appearance/domain/AppBackgroundImageStore.kt
```

该类负责背景图片导入、压缩和清理：

- 从用户选择的 `Uri` 读取图片。
- 使用系统 `BitmapFactory` 解码，无新增 Coil / Glide 等依赖。
- 降采样并限制最大边长，避免启动和渲染阶段内存压力过高。
- 将图片压缩保存到 App 私有目录：`filesDir/theme/background`。
- 成功导入新图片后，清理旧的私有背景文件。
- 删除文件前校验 canonical path，避免误删非本功能拥有的文件。

当前导入结果只返回私有图片路径，不再自动取色。

### 3.2 图片取色开关

用户明确要求“不要导入后自动抽取，而是给一个开关让用户在从图片取色还是当前的系统取色中间选一个”。因此当前流程为：

```text
选择图片
→ 仅导入并启用背景
→ 默认仍使用系统 / 当前主题取色
→ 用户手动打开“图片取色”
→ 从私有图片提取 seed color
→ 复用 ThemeColorScheme.CUSTOM 链路生成主题色
```

`SettingsViewModel.updateAppBackgroundImageColorEnabled(...)` 负责该流程：

- 关闭时：`appBackgroundImageColorEnabled = false`，并恢复 `ThemeColorScheme.DEFAULT`。
- 开启时：要求已存在背景图片。
- 如果已有合法 seed，则复用。
- 如果没有 seed，则通过 `AppBackgroundImageStore.extractSeedColorHex(path)` 提取。
- 成功后写入 `customThemeColorHex`，并把 `themeColorScheme` 设为 `CUSTOM`。

当用户手动修改主题色或自定义主题色时，会自动关闭图片取色，避免两个主题来源互相覆盖。

## 五、主背景层与页面透明化

### 5.1 根层背景图片渲染

`MainActivity` 新增根层背景渲染：

```text
CalendarAssistantStyleTheme
→ CompositionLocalProvider(background bitmap / root size)
→ Box(background = MaterialTheme.colorScheme.background)
→ AppBackgroundLayer
→ NavHost
```

当前 `AppBackgroundLayer` 接收已解码的 `ImageBitmap`，不再自己重复解码文件：

```kotlin
AppBackgroundLayer(
    enabled = settings.appBackgroundImagePath.isNotBlank(),
    imageBitmap = appBackgroundBitmap,
    blurEnabled = settings.appBackgroundWallpaperBlurEnabled,
    modifier = Modifier.fillMaxSize()
)
```

图片使用 `ContentScale.Crop` 填充全屏。`blurEnabled` 只受 `壁纸模糊` 控制。

### 5.2 背景图解码上提

为支持 glass 内部复用壁纸，本轮把背景图片解码从 `AppBackgroundLayer` 内部上提到 `MainActivity` 根组合区域：

```kotlin
val appBackgroundBitmap by produceState<ImageBitmap?>(key1 = settings.appBackgroundImagePath) {
    value = if (settings.appBackgroundImagePath.isBlank()) null else withContext(Dispatchers.IO) {
        BitmapFactory.decodeFile(settings.appBackgroundImagePath)?.asImageBitmap()
    }
}
```

并通过 CompositionLocal 提供给子树：

```text
LocalAppBackgroundWallpaperBitmap
LocalAppBackgroundRootSize
```

这使得 glass 组件能够绘制一份与全屏背景对齐的“背景副本层”。

### 5.3 避免页面遮挡背景

为了让根背景可见，本轮调整了多个页面容器：

- `NavHost` 不再额外覆盖 `MaterialTheme.colorScheme.background`。
- 首页存在背景图时，`HomePage` 的 `Scaffold` 使用透明背景。
- `PushSlideLayout` 新增 `contentContainerColor`，背景模式下传入 `Color.Transparent`。
- 设置详情页的公共 `Scaffold` 和 topbar 在背景模式下使用透明容器。
- 归档页被并入设置页公共 scaffold/topbar，不再拥有特殊 topbar。

## 八、真实 MIUI glass 模糊尝试

### 8.1 不能直接模糊 Card 本体

用户反馈打开 `MIUI 风格测试` 后，卡片只是更透明，没有真正模糊。排查后确认旧实现只是改变 palette 和透明度，没有给 glass 卡片绘制真实模糊层。

本轮确认不能简单对 `Card` 本体加 `BlurEffect`：

- 如果对整个 Card 应用 `BlurEffect`，文字、图标、开关也会一起模糊。
- 需要先绘制一个背景副本层并模糊，再叠加 tint，最后绘制前景内容。

### 8.2 当前实现方式

`AppBackgroundGlassSurface` 被改造成三层结构：

```text
Box(shape clip + border)
├─ AppBackgroundBlurredBackdrop   // 只在 MIUI 测试开启时绘制
├─ 半透明 tint 层
└─ 前景内容                       // 文字、图标、开关不模糊
```

`AppBackgroundBlurredBackdrop` 的做法：

- 读取 `LocalAppBackgroundWallpaperBitmap`。
- 读取 `LocalAppBackgroundRootSize`。
- 用 `onGloballyPositioned` 获取当前卡片在 root 中的位置。
- 在卡片内绘制一张 root 尺寸的背景图副本。
- 根据卡片坐标负向 offset，使卡片内看到的背景区域与全屏背景对齐。
- 对该背景副本层应用 `BlurEffect(28.dp)`。

该方案是“背景图副本 + 对齐裁剪 + RenderEffect 模糊”的近似实现，不是 MIUIX 那套真正的 backdrop capture 管线。

### 8.3 MIUIX 项目调研结论

用户提供过：

```text
C:\Users\ASUS\Downloads\miuix-main.zip
```

本轮解压到临时目录后查看，发现其 `miuix-blur` 是 Kotlin Multiplatform 模块，依赖 `miuix-shader`，通过 `rememberLayerBackdrop`、`layerBackdrop`、`drawBackdrop`、`textureBlurEffect`、`RuntimeShader` 和 `RenderEffect` 组合实现真正的 backdrop 模糊。

结论：

- 直接接入 MIUIX blur 成本较高。
- 它不是一个简单 Android View API，而是 KMP + shader + layer backdrop 管线。
- 本轮只做项目内低风险实验，没有引入第三方依赖。

## 九、当前已知问题

### 9.1 用户实测只有少数板块可见模糊

装机后用户反馈：

```text
只看到侧边栏里面的最上面主题设置、关于软件、软件更新这个板块 card 有模糊；
其它地方一点模糊都没有。
```

随后本窗口只读排查确认：

- `AppBackgroundGlassSurface(` 的调用点非常少。
- 当前真实模糊容器只接入了少数位置：主题设置页部分卡片、开发者页本地 `SettingsCard`、侧栏 `SidebarGlassCard`、底栏浮动容器。
- 大量设置页仍直接使用普通 `Card(colors = CardDefaults.cardColors(containerColor = MaterialTheme.colorScheme.surfaceContainerLow))`。
- 这些普通卡片只会吃到背景模式色板，表现为“半透明/深浅色变化”，不会绘制 `AppBackgroundBlurredBackdrop`。

排查到仍有普通卡片的页面包括但不限于：

```text
ConfigEditorPage.kt
BottomBarEditorPage.kt
WidgetSettingsPage.kt
BackupSettingsPage.kt
WeatherSettingsPage.kt
WeatherDetailPage.kt
AiSettingsPage.kt
TimeTableEditorScreen.kt
PreferenceSettingsPage.kt
LaboratoryPage.kt
RegexRuleEditorPage.kt
ScheduleSettingsPage.kt
```

### 9.2 侧栏只有顶部板块明显模糊的疑点

代码上 `SettingsSidebar` 的多个分组都已经走 `SidebarGlassCard(glassMode)`，理论上都应进入 `AppBackgroundGlassSurface`。

但用户实测只有顶部板块明显有模糊，说明可能还有以下问题之一：

- 背景副本层坐标对齐在侧栏滚动区域内存在偏差。
- 下方卡片区域对应背景本身细节较少，视觉上不明显。
- 某些卡片外层或内部 tint 过强，遮蔽了模糊效果。
- 当前近似实现只复制静态壁纸，不是真正捕获卡片背后的完整已绘制内容。

该问题尚未继续深入修复。

### 9.3 本轮未完成的统一抽象

本轮为了快速验证效果，直接把部分 `Card` 替换为 `AppBackgroundGlassSurface`，没有完成全设置页统一抽象。

推荐后续改法：

- 新增统一组件，例如 `SettingsGlassCard` 或 `BackgroundAwareCard`。
- 默认无背景时内部仍使用 Material `Card`。
- 背景模式且 MIUI 测试开启时使用 `AppBackgroundGlassSurface`。
- 背景模式但 MIUI 测试关闭时使用普通半透明 glass 色板。
- 分批替换所有设置页中的 `Card(surfaceContainerLow)`，不要继续零散 patch。

## 十二、后续接手建议

### 12.1 先统一 glass 卡片抽象

不要继续在各页面里手工寻找 `Card` 并零散替换。建议先做统一组件：

```kotlin
@Composable
fun SettingsGlassCard(
    modifier: Modifier = Modifier,
    shape: Shape = RoundedCornerShape(16.dp),
    content: @Composable ColumnScope.() -> Unit
)
```

组件内部根据 CompositionLocal 判断：

- 无背景模式：普通 Material `Card(surfaceContainerLow)`。
- 有背景但未开 MIUI 测试：半透明 glass tint。
- 有背景且 MIUI 测试：`AppBackgroundGlassSurface`。

然后优先替换设置页内所有同风格卡片。

### 12.2 优先排查侧栏下方卡片不明显问题

虽然代码上侧栏各分组都走 `SidebarGlassCard`，但用户实测只有顶部组明显模糊。后续应先在 `AppBackgroundBlurredBackdrop` 中验证：

- `LocalAppBackgroundRootSize` 是否稳定非零。
- 各卡片 `positionInRoot()` 是否正确。
- 滚动后位置是否更新。
- root 尺寸背景副本在 `ContentScale.Crop` 下是否和真实背景一致。
- tint alpha 是否过高导致模糊被遮住。

### 12.3 再批量替换设置页普通 Card

### 12.4 背景闪现问题仍未从根上解决

用户此前反馈 App 从后台回来时会先显示默认背景，再出现图片背景。当前只是把解码上提并复用 bitmap，仍是 Compose 内异步 decode，未引入 Coil，也未做 Splash readiness。

如后续要根治，建议单独立项：

- 方案一：引入 Coil Compose，由图片加载库托管内存/磁盘缓存。
- 方案二：把背景 decode 状态上提到启动准备阶段，配合 SplashScreen 控制首帧。

本轮未继续处理该问题。
