# 2026-04-13 至 2026-04-21 v47 开发记录

> 本文是对 2026-04-13 至 2026-04-21 期间 v47-v57 多篇开发记录的合并复盘。原始记录仍保留，本文用于统一说明：为什么要做这轮重复日程修复与架构升级，过程中尝试过哪些办法，哪些办法被保留，哪些方案最终放弃，以及最终形成的架构基线。

## 一、这轮工作的起点

这一轮工作的起点看起来是“修重复日程”，但实际很快扩展成了一次架构升级。

原因是重复日程不是普通的增删改查。一个重复事件至少包含以下几类对象和状态：

- 重复父事件：保存 RRULE、系列标题、系列时间等主信息。
- 重复实例：由父事件按规则展开出来的某一次 occurrence。
- 单次异常：只修改某一次时，需要从原系列中分离出 exception。
- 单次删除：只删除某一次时，需要在父事件上记录排除，或写入 canceled exception。
- 此后修改/删除：需要截断旧系列，再创建或保留新的后半段系列。
- 业务状态：完成、检票、取件、归档、Undo、提醒、胶囊展示等。

旧实现的问题是：重复日程的 UI 操作、系统日历同步、通知动作、Undo、课程表等链路各自处理一部分语义，导致同一件事在不同入口下行为不一致。

典型问题包括：

- App 内修改重复日程单次实例后，系统日历出现“修改后的单例 + 原始重复实例”的 Twins 问题。
- 通知栏点击“已完成”后，重复实例短暂完成，随后又被反向同步覆盖复活。
- 系统日历多日历同步时，一个日历的重复系列可能被另一个日历的 stale 清理误删。
- “今日日程”和“全部日程”展示口径不一致，全部页显示父占位而不是当天实例。
- 课程表有独立模型，实际却也需要重复日程的 this/future/all 语义，形成双模型负担。
- UI、Service、Receiver 直接调用仓库或数据库，副作用分散，难以判断某次操作到底触发了哪些同步、提醒和胶囊刷新。

因此本轮不是简单修 bug，而是围绕重复日程问题逐步做了三件事：

1. 先把重复日程语义补齐并止血。
2. 尝试 Provider-only，让系统日历成为唯一事实源。
3. 在实测后放弃 Provider-only，最终转向 Room + Provider + 系统提醒，并把调用链升级为 Center/API/Node。

## 二、第一阶段：重复日程语义补齐与同步止血

对应原始记录：v47、v49、v50 语义定稿、v51。

### 2.1 为什么先补重复日程语义

旧版本重复日程操作能力不足，主要只有“仅修改此事件”和“修改所有事件”一类简化入口，删除也没有重复日程专属范围选择。

这与用户对标准日历的预期不一致。用户编辑或删除重复事件时，通常需要三种范围：

- 仅此一次。
- 此事件及以后。
- 整个系列。

因此本轮先将重复日程操作补齐为 1-6 操作：

| 类型 | 操作 | 目标语义 |
|---|---|---|
| 编辑 | 编辑仅此一次 | 当前实例分离为 exception，其他实例不变 |
| 编辑 | 编辑此及以后 | 截断旧系列，当前实例开始使用新系列 |
| 编辑 | 编辑全部 | 整个重复父系列统一更新 |
| 删除 | 删除仅此一次 | 当前实例被排除，其他实例保留 |
| 删除 | 删除此及以后 | 首实例等价删整组，非首实例截断旧系列 |
| 删除 | 删除全部 | 删除父系列及所有实例/异常 |

### 2.2 第一轮采用的办法：EXDATE + exception

为了解决“只改/只删某一次”的问题，采用了与标准日历一致的思路：

- 对“仅此一次编辑”：把当前实例从父系列中排除，再创建一个独立 exception 事件。
- 对“仅此一次删除”：把实例 key 写入父事件的排除列表，或在 Provider 中写 canceled exception。
- 对“完成重复实例”：不再直接修改重复子实例，而是将该实例转成完成态 exception。
- 对“Undo 重复实例完成”：删除 exception，并从父事件排除列表中移除该实例。

这样做的原因是，重复实例本身并不是稳定的独立实体。它通常是父事件 + RRULE 展开出来的临时结果。如果直接修改一个展开实例的本地状态，下一次系统日历反向同步可能重新按父规则展开，把本地修改覆盖掉。

因此最终确立了一个原则：

> 重复实例的单次状态变化必须落到“父事件排除 + exception”这类稳定结构上，不能只改临时实例。

### 2.3 修复 Twins 问题

Twins 问题的表现是：App 内修改某个重复实例后，同一天同时出现两个事件：

- 修改后的单例。
- 原始重复系列展开出的实例。

根因是系统日历中的父事件没有记录 EXDATE，系统仍然会按 RRULE 在该日期生成原始实例。

当时采取的办法：

- 在父事件上追加 EXDATE。
- 对 MIUI/HyperOS 等不会自动补 EXDATE 的 Provider 做手动补偿。
- 对托管事件写入标记，避免反向同步把 App 自己创建的重复事件再次导回。

这个办法被保留下来，成为后续重复实例操作的基础。

### 2.4 修复通知完成后复活

旧逻辑中，通知栏“已完成”会直接把重复子实例标记为完成。问题是该子实例仍属于重复系列，系统日历里对应 occurrence 并没有变化。反向同步后，系统数据会重新覆盖本地完成态。

修复方式是把“完成重复实例”也当作“仅此一次 exception”处理：

1. 父事件记录排除该 instanceKey。
2. 移除原重复子实例。
3. 创建新的完成态 exception。
4. 触发同步和提醒重排。

这样完成态变成了稳定实体，不会被下一次展开覆盖。

### 2.5 修复跨日历误删与 source 漂移

系统日历反向同步存在多日历场景。旧实现的 stale 清理没有严格按日历前缀隔离，导致 calendar 4 同步时可能误清 calendar 1 的重复系列。

修复方式：

- stale 清理按 calendarPrefix 限定范围。
- source 漂移时保留已有 system_recurring 来源，不再被覆盖成 sync_calendar。
- 反向同步实例数量超过上限时不再中断全部流程，而是允许部分同步，并跳过删除阶段，避免误删。

这部分解决的是“同步安全性”：宁可少清理，也不能错误删除用户数据。

### 2.6 统一展示口径

旧的“全部日程”页会展示重复父事件占位，而“今日日程”展示真实实例，用户感知为某些日程在全部页缺失。

修复方式：

- 全部页不展示重复父事件。
- 展示重复子实例与普通事件。
- 移除父事件“下次时间”展示，避免和实例视图混淆。

这个规则后来也被写入 API 边界文档：All 页展示过滤应在过滤后的集合上执行，重复父事件不进入展示列表。

### 2.7 Undo 统一化

旧 Undo 主要依赖 completedAt 等分散字段判断，容易出现不同动作撤销逻辑不一致。

本轮定稿了 60 秒 UndoToken 模型：

```kotlin
data class UndoToken(
    val tokenId: String,
    val targetKey: String,
    val actionType: String,
    val beforeSnapshotId: String,
    val afterSnapshotId: String,
    val expireAtMillis: Long
)
```

最终原则：

- 触发终态动作时创建新数据，并保存可回退的 before/after 关系。
- 60 秒内可撤销。
- Undo 必须按 actionType 精确消费，避免把“完成”撤销成“检票”或反过来。
- 对重复实例，Undo 的核心是删除本次 exception，并恢复父事件排除关系。
- 如果系列在撤销前已经被删除或切分，则撤销失败，提示实例不可恢复。

### 2.8 阶段结论

第一阶段保留下来的有效做法是：

- 重复实例单次变化必须转为 exception。
- 单次删除必须体现在父系列排除或 canceled exception 上。
- this/future/all 是重复日程的基础操作语义，不能由 UI 临时拼接。
- Undo 必须统一 token 化，不能散落在各个动作里。
- 同步清理必须宁可保守，不能跨日历误删。

## 三、第二阶段：Provider-only 尝试

对应原始记录：v50 Provider-only 宪法、v50 上机测试、v52 开发复盘。

### 3.1 为什么会尝试 Provider-only

重复日程天然属于系统日历 Provider 的能力范围。系统日历已经提供：

- RRULE。
- Instances 展开。
- Exception。
- STATUS_CANCELED。
- ORIGINAL_INSTANCE_TIME。
- this/future/all 语义的底层支持。

因此当时的想法是：与其在 App 内维护一套复杂的重复日程结构，不如让 Calendar Provider 成为唯一事实源。

Provider-only 的目标是：

- App 不再维护一套独立事件主存。
- 所有事件事实都以 CalendarContract 为准。
- App 的 UI 读取 Provider 实例流。
- 重复日程 this/future/all 全部以 Provider 语义为主。
- `SYNC_DATA1/SYNC_DATA2` 用于保存 App 元数据和 appEventId。

当时也为此制定了若干硬规则：

- `SYNC_DATA2` 固定保存 appEventId(UUID)。
- `SYNC_DATA1` 保存 ProviderSyncData 状态元数据。
- 所有 Provider 写操作走 sync-adapter 参数。
- 写前通过 CalendarBootstrapper 保证本地可写日历存在。
- 有 RRULE 的主事件只写 DTSTART + DURATION，不写 DTEND。
- all-day splitFuture 使用 UTC yyyyMMdd UNTIL。
- all-day 完成只改状态，不改 DTEND/DURATION。

### 3.2 Provider-only 阶段采取的办法

这一阶段做过的主要尝试包括：

- 新增 Provider 侧 Port/Node：EventStore、RecurringEditor、CompletionPort、MetaPort、ArchivePort。
- 将单例、重复、exception、归档、完成、元数据尽可能写入 Calendar Provider。
- 删除旧 CalendarSyncManager，统一走 CalendarSyncManagerV2。
- 移除周期性反向同步广播，保留 CalendarContentObserver + WorkManager。
- 尝试让 EventMetaStore 退化为可重建缓存。
- 将规则系统从 Room 表迁出，改为文件规则存储。
- 对 Provider 操作做上机冒烟测试：单例 CRUD、重复创建、this exception、delete this、future split、delete all 等。

从底层 Provider 操作看，很多单点能力确实跑通了。v50 阶段 Provider 冒烟脚本曾达到 9/9 通过。

### 3.3 Provider-only 暴露的问题

实机使用后发现，Provider-only 不是单点操作不通，而是作为 App 唯一事实源不稳。

关键问题有四类。

#### 3.3.1 tag 和业务状态不是 Calendar 标准字段

App 需要保存很多系统日历没有的业务状态：

- tag：取件、课程、便签、出行、普通日程等。
- 归档态。
- 完成态。
- 检票态。
- 取件/取餐/用车/取票等规则动作状态。
- Undo 窗口。
- 胶囊展示状态。

这些状态不能可靠地映射到标准 Calendar 字段。

可选方案是写 ExtendedProperties 或 SYNC_DATA，但它们在不同 ROM、不同账号类型、不同系统日历实现上并不稳定。

#### 3.3.2 ExtendedProperties 写入可能拖垮主事件写入

当时发现 `tag` 依赖 CalendarContract.ExtendedProperties。部分非本地账号日历可能拒绝写 ExtendedProperties。

如果 Events、Reminders、ExtendedProperties 在同一个批处理中提交，其中一步失败可能导致整批失败，最终表现为：

> 用户点击新建，弹窗消失，但事件没有创建出来。

这说明业务元数据不能和事件主写入强耦合。主事件写入必须优先成功，tag 等业务状态应是独立、可降级、可恢复的。

#### 3.3.3 Provider 实例视图无法稳定替代本地 parent 模型

系统 Provider 查询常给的是实例视图，而不是 App 内稳定父对象。

重复日程 this/future/all 需要定位：

- 父系列。
- 当前实例 begin。
- 是否首实例。
- 原 RRULE。
- 已有 exception/exdate。

Provider-only 下，如果 UI 或业务链路仍依赖本地 parent 记录，就会出现“未找到对应的重复系列信息”。如果完全不保留本地 parent，又需要在每个入口临时反查 Provider 结构，复杂度和兼容性风险都很高。

#### 3.3.4 前端业务态没有独立落点，恢复和回滚困难

归档、Undo、完成、检票等状态不是单纯日历事实。它们更像 App 的业务视图。

如果全部塞进 Provider：

- 某些字段可能被系统日历修改或忽略。
- 某些账号可能不同步或拒写。
- App 很难判断哪些状态是自己写的，哪些是外部日历变化。
- Undo 这类短时状态不适合写入系统日历作为唯一事实。

### 3.4 为什么最终放弃 Provider-only

最终放弃 Provider-only，不是因为 Provider 的重复日程能力不可用，而是因为它不适合作为本 App 的唯一事实源。

结论如下：

1. Provider 适合承载日历事实：时间、地点、标题、RRULE、exception、系统互通。
2. Provider 不适合承载全部 App 业务状态：tag、归档、Undo、检票、规则动作、胶囊状态。
3. ExtendedProperties/SYNC_DATA 可以作为 best-effort 辅助，但不能成为唯一可靠存储。
4. 主事件写入不能被 tag 等扩展字段失败拖垮。
5. 重复实例语义可以参考 Provider/Etar，但 App 仍需要自己的稳定业务投影。

因此 Provider-only 被放弃，转向 Room + Provider + 系统提醒。

## 四、最终数据架构：Room + Provider + 系统提醒

对应原始记录：v52 开发复盘、v55-v57。

### 4.1 新架构的角色分工

最终确定的架构不是“只用 Room”，也不是“只用 Provider”，而是三者分工：

| 层 | 职责 |
|---|---|
| Room | App 主存与业务状态源，保存 tag、归档、完成、检票、Undo、课程投影等 |
| Calendar Provider | 系统日历互通层，负责时间、RRULE、exception、系统日历展示和外部修改回流 |
| 系统提醒 | 执行层，负责 Alarm、普通通知、胶囊通知、取消和重排 |

### 4.2 写入原则

最终写入原则：

1. 业务写入口统一进入 App 的写链路。
2. Room 保存稳定业务状态。
3. Provider 写入作为系统互通动作，失败要可感知、可恢复。
4. tag 等扩展字段可尝试写 Provider，但只能 best-effort，失败不能回滚事件主写入。
5. 写入完成后统一发出变更事件，触发提醒重排和胶囊刷新。

### 4.3 读取原则

读取侧采用“业务状态 + Provider/同步投影”的方式：

- App 主界面以 Room 中的事件与状态为稳定来源。
- 系统日历变化通过反向同步进入 Room 投影。
- UI 不直接依赖 Provider 实例视图做业务判断。
- 重复实例的展示由事件模型统一展开和过滤。

### 4.4 为什么这个方案被保留

这个方案保留的原因是它同时满足三类要求：

- 能和系统日历互通：Provider 仍负责日历事实和重复规则。
- 能保证 App 业务态可靠：Room 保存不可丢的业务状态。
- 能控制提醒与胶囊：系统提醒作为独立执行层，由统一事件链驱动。

相比 Provider-only，它牺牲了一点“理论上的单事实源纯粹性”，但换来了稳定性、可恢复性和可调试性。

## 五、第三阶段：操作/API 边界收口

对应原始记录：v53、v54。

### 5.1 为什么要做 API 边界

Provider-only 放弃后，项目又回到“App 自己有主存和业务状态”的模式。但如果 UI、Service、Receiver 都能直接调用仓库或数据库，问题仍然会复发：

- 某个入口可能只改 Room，不触发 Provider 同步。
- 某个入口可能只触发通知，不更新胶囊。
- 某个 Receiver 可能绕过 Undo 或规则动作。
- 页面里直接写仓库会让业务语义散落在 UI 中。

因此必须先定义操作边界。

### 5.2 前端操作与后端操作划分

本轮明确了两类操作：

#### 前端操作

只改变 UI 临时状态，不改变业务事实。例如：

- Tab 切换。
- 搜索框文本。
- 弹窗显隐。
- 手势偏移。
- 焦点和键盘状态。
- 局部展开态。

这类操作允许留在页面或组件内部。

#### 后端操作

会改变业务事实、持久化数据或触发外部副作用。例如：

- 新增、编辑、删除日程。
- 重复日程 this/future/all 操作。
- 完成、检票、撤销。
- 归档、恢复、清空归档。
- 同步、提醒、胶囊刷新。
- OCR/短信识别后入库。
- 设置写入与副作用联动。

这类操作必须走 API/Center，不允许页面、Service、Receiver 直接写仓库或数据库。

### 5.3 写操作 API 化

写链路收口为：

```text
调用方 -> OperationApi -> Store/Node
```

当时新增并迁移了：

- `ScheduleOperationApi`。
- `SettingsOperationApi`。
- 之后又扩展出 `BackupOperationApi`、`WeatherOperationApi`、`IngestCommandApi` 等。

迁移范围包括：

- MainViewModel。
- SettingsViewModel。
- EventActionReceiver。
- FloatingScheduleService。
- TextAccessibilityService。
- SmsReceiver / SmsNotificationListenerService / SmsContentObserver。
- EdgeBarService。
- CalendarReverseSyncWorker。

### 5.4 读操作 API 化

写链路收口后，读链路也需要对称收口。否则 ViewModel 或页面仍会直接读取仓库内部 Flow，边界仍不完整。

因此新增：

- `ScheduleQueryApi`。
- `SettingsQueryApi`。
- 后续扩展出 Home、SettingsTransform、EventAction、NotificationPresentation、AlarmRouting、CapsuleRouting、ScheduleInsights、DailySummary 等 Query API。

读 API 化的目的不是改变展示逻辑，而是让 UI 层不再直接感知底层仓库和数据库。

### 5.5 架构护栏

为了防止后续代码回流旧链路，新增架构护栏任务：

- `checkArchitectureGuardrails`。
- `updateArchitectureGuardrailsBaseline`。

护栏禁止 UI/Service/Receiver/Worker 等调用方直接使用：

- `AppRepository.getInstance(...)`。
- `(applicationContext as App).repository`。
- `AppDatabase.getInstance(...)`。

这一步很重要。没有自动护栏，重构很容易在后续小修小补中被绕开。

## 六、第四阶段：Center/API/Node 架构落地

对应原始记录：center-port-node 架构定稿、v55、v56、v57。

### 6.1 最初的 Center-Port-Node 设想

最初定稿的架构是：

```text
UI/Service/Receiver -> Center -> Port -> Node
```

各层职责：

- Center：业务决策和流程编排。
- Port：能力边界接口。
- Node：基础设施和执行层，例如 Room、Provider、通知、胶囊、Worker、Receiver。

这个设计的目标是解决两类问题：

- 上层调用方不再知道底层实现。
- 业务流程不再散落在 UI、Service、Receiver、Repository 中。

### 6.2 为什么要拆 AppRepository 巨石

当时的 AppRepository 承担了太多职责：

- 日程 CRUD。
- 重复日程操作。
- 归档。
- 同步。
- 备份导入导出。
- 完成/检票/Undo。
- 设置副作用。
- 胶囊刷新。
- 提醒调度。

这导致任何修改都可能影响多个域，且很难确认某个入口是否完整触发了同步、提醒和 UI 刷新。

因此拆分方向是：

- 上层只调用 Center/API。
- 数据主链下沉到 StoreRootNode。
- 具体能力拆成 Store*Node。

### 6.3 Store 主链收口

本轮最终形成的数据主链：

```text
Center
  -> Core API(operation/query)
    -> StoreDispatcher
      -> StoreRootNode
        -> Store*Node
          -> Room / JSON / CalendarSyncManager(V2) / System Calendar
```

关键变化：

- `AppRepository` 类型下线。
- `StoreCoordinator` 过渡后重命名为 `StoreRootNode`。
- `Store*Support` 统一重命名为 `Store*Node`。
- `StoreDispatcher` 对外实现核心 API，内部调用 `StoreRootNode`。
- Store 域日志、注释、命名清理旧 `AppRepository` 语义。

### 6.4 Center 逐步落地

本轮陆续落地或升级了多个中心：

| Center | 职责 |
|---|---|
| ScheduleCenter | 日程动作、查询、重复实例、业务入口编排 |
| SyncCenter | 同步开关、手动同步、反向同步、来源日历管理 |
| ReminderCenter | 提醒重排、胶囊刷新桥接、通知互斥与路由决策 |
| CapsuleCenter | 胶囊状态控制与刷新 |
| RecognitionCenter | OCR、文本、多模态识别编排 |
| ContentIngestCenter | 识别/SMS 等内容源统一入库，Actor 串行消费 |
| BackupCenter | 备份导入导出独立编排 |
| RuntimeCenter | App 启动、开机恢复、保活、早晚报、侧边栏、网速监控等运行时编排 |
| PermissionCenter | 日历、悬浮窗、通知监听等权限判断收口 |
| NotificationCenter | 普通通知构建和按钮编排 |
| FloatingCenter | 悬浮窗和侧边栏启动、权限门禁 |
| RuleCenter | 规则实验室读写与状态读取 |

### 6.5 事件化链路

为了解决“写入后谁来刷新提醒和胶囊”的问题，本轮引入了轻量 DomainEventBus。

关键事件包括：

- `recognition.completed`。
- `recognition.failed`。
- `ingest.succeeded`。
- `ingest.failed`。
- `schedule.changed`。
- `capsule.refresh.requested`。

最终形成的关键链路：

```text
RecognitionCenter
  -> recognition.completed
    -> ContentIngestCenter Actor
      -> StoreDispatcher 写入
        -> schedule.changed
          -> ReminderCenter
            -> 提醒重排 + 胶囊刷新
```

这样 Home、Note、Floating、Accessibility 四条识别入口不再各自 addEvent，而是统一走 ingest 主链。

### 6.6 提醒与胶囊联动收口

旧逻辑中，提醒、普通通知、胶囊、MIUI 岛分支散落在 NotificationScheduler、AlarmReceiver、CapsuleStateManager、AppRepository 中。

本轮收口方式：

- `schedule.changed` 由 `ReminderCenter` 订阅。
- 提醒对齐做 250ms 聚合，避免批量写入高频 cancel/schedule。
- 对 CREATE/RESTORE/UPDATE/BULK/DELETE/ARCHIVE 做动作分流。
- 空 eventIds 的 BULK(import/sync/system) 做全量兜底。
- 胶囊刷新通过 `capsule.refresh.requested` 桥接。
- AlarmReceiver 中胶囊路由、MIUI 岛、普通通知判断逐步下沉到 ReminderCenter/NotificationCenter。

最终目标是：Receiver 只做 Android 壳层路由，业务决策在 Center。

### 6.7 为什么最后简化为 Center -> API -> Node

原始设计有 Port 层，但后续发现部分 Port 只是纯转发壳，增加了命名和调用层级，却没有提供额外语义。

因此 v57 按最小改动方案移除纯转发 Port 壳层，最终收敛为：

```text
UI / Service / Receiver / Worker
  -> Center
    -> Core API(operation/query contracts)
      -> StoreDispatcher / 其他 API 实现
        -> Node
```

放弃“长期保留纯 Port 壳层”的原因：

- 当前项目更需要清晰边界，而不是过多中间层。
- 纯转发接口没有独立业务含义，反而增加维护成本。
- API 合约已经能表达调用边界。
- Node 已经承载稳定能力边界。

因此最终保留的是 API 语义层，而不是所有 Port 形态。

## 七、第五阶段：课程并入重复日程

对应原始记录：v52 课程并入日程定稿、v57。

### 7.1 为什么课程必须并入日程

课程表一开始是独立模型和独立存储，但实际使用中，课程天然就是重复日程：

- 每周/单双周重复。
- 有学期起止范围。
- 可以修改某一次上课时间。
- 可以删除某一次课程。
- 应该能进入系统日历。
- 应该能和普通日程一起展示、提醒、归档、同步。

如果课程继续保留独立模型，就会导致：

- 课程一套重复语义，普通日程一套重复语义。
- 课程同步单独展开，普通重复日程同步按 RRULE。
- UI 要分别处理 Course 和 Event。
- “修改本次课程”无法自然复用重复日程 exception 语义。

因此最终决定：课程不再作为独立业务模型，而是统一为事件。

### 7.2 最终课程模型

课程统一为：

```text
MyEvent(tag = course)
```

语义如下：

- 新建课程：写入重复父事件。
- 更新全部：修改重复父事件。
- 更新本次：detach 当前重复实例，形成单次实例。
- 删除本次：写入父事件 excludedRecurringInstances，或删除已分离实例。
- 删除整门课：删除父系列及分离实例。
- 课程分离实例允许同步到系统日历。

### 7.3 课程 description 规则

课程 description 保留可读文本，不引入新的结构化 payload：

```text
【课程】周一上午一二节|老师名|单双周
```

其中：

- 第 1 段：课程时间映射文案。
- 第 2 段：教师名称。
- 第 3 段：周类型。

标准字段仍然以事件字段为准，description 只承载可读辅助信息。

### 7.4 同步链路变化

v57 删除课程专用同步分支：

- 删除课程展开同步路径。
- 删除课程哈希字段。
- 正向同步统一按事件同步。
- 课程重复系列通过 recurring master/instance 进入系统日历。

最终课程与普通重复日程共享同一条同步通道。

## 八、本轮明确放弃的方案

### 8.1 放弃 Provider-only 作为唯一事实源

放弃原因：

- App 业务态无法可靠落到 Calendar Provider。
- ExtendedProperties/SYNC_DATA 在不同账号和 ROM 上不稳定。
- tag 写入失败可能拖垮事件创建。
- Provider 实例视图不能稳定替代 App 业务模型。
- Undo、归档、检票、胶囊等状态需要独立可控存储。

保留部分：

- Provider 的重复日程语义仍被采用。
- this/future/all、exception、EXDATE、canceled exception 继续作为系统同步语义。
- Provider 继续负责系统日历互通。

### 8.2 放弃旧 AppRepository 巨石入口

放弃原因：

- 职责过多。
- 修改影响面不可控。
- UI/Service/Receiver 容易绕过完整副作用。
- 难以保证提醒、同步、胶囊、Undo 语义一致。

替代方案：

- `StoreDispatcher -> StoreRootNode -> Store*Node`。
- 上层只通过 Center/API 调用。

### 8.3 放弃课程独立模型

放弃原因：

- 课程本质是重复日程。
- 独立模型会造成重复语义、同步、展示和编辑的双轨。
- 单次课程修改/删除应直接复用重复日程 exception/excluded 语义。

替代方案：

- `MyEvent(tag=course)`。
- 课程 UI 从事件投影读取。
- 同步统一走重复日程主链。

### 8.4 放弃 UI/Receiver 直连仓库或数据库

放弃原因：

- 会把业务事实修改散落到调用层。
- 容易漏掉提醒、胶囊、同步、Undo 等副作用。
- 不利于回归和架构约束。

替代方案：

- 后端操作必须走 Center/API。
- 读操作走 Query API。
- 写操作走 Operation API。
- 架构护栏持续检查。

### 8.5 放弃长期保留纯转发 Port 壳层

放弃原因：

- 纯转发没有增加语义。
- 增加文件数量和调用层级。
- 当前 API 合约已经足够表达边界。

替代方案：

- `Center -> API -> Node`。
- 仅在确实需要隔离实现时保留接口边界。

## 九、最终架构图

### 9.1 总体调用方向

```text
UI / ViewModel / Service / Receiver / Worker
  -> Center
    -> Core API (Operation / Query)
      -> StoreDispatcher 或对应 API 实现
        -> Node
          -> Room / JSON / Calendar Provider / Notification / Capsule / Worker
```

### 9.2 日程写入链路

```text
用户操作 / 通知动作 / 识别入库
  -> ScheduleCenter / ContentIngestCenter
    -> ScheduleOperationApi / IngestCommandApi
      -> StoreDispatcher
        -> StoreRootNode
          -> StoreEventNode / StoreEventActionNode / StoreCalendarSyncNode
            -> Room 主存
            -> CalendarSyncManagerV2
              -> System Calendar Provider
          -> schedule.changed
            -> ReminderCenter
              -> 提醒重排 + 胶囊刷新
```

### 9.3 重复日程语义链路

```text
重复实例操作(this/future/all)
  -> ScheduleCenter
    -> ScheduleOperationApi
      -> StoreDispatcher
        -> StoreRootNode
          -> StoreEventActionNode
            -> 父事件 / exception / excludedRecurringInstances
            -> CalendarSyncManagerV2
              -> RRULE / EXDATE / exception / canceled instance
```

### 9.4 识别入库链路

```text
Home / Note / Floating / Accessibility
  -> RecognitionCenter
    -> recognition.completed(ingestRequested=true)
      -> ContentIngestCenter Actor
        -> ingest recognized events
          -> StoreDispatcher 写入
            -> schedule.changed
              -> ReminderCenter
```

### 9.5 课程链路

```text
Course UI Action
  -> MainViewModel
    -> ScheduleCenter
      -> ScheduleOperationApi
        -> StoreDispatcher
          -> StoreRootNode
            -> MyEvent(tag=course, isRecurringParent=true)
              -> recurring master / instance / excluded
                -> CalendarSyncManagerV2
                  -> System Calendar
```

## 十一、后续开发必须遵守的规则

### 11.1 重复日程规则

1. 不允许直接修改临时重复实例来表达单次变化。
2. 仅此一次编辑必须走 exception。
3. 仅此一次删除必须走 excluded/EXDATE/canceled exception。
4. 此及以后必须走截断旧系列和新系列/后续语义。
5. 完成重复实例必须按单次 exception 处理。
6. Undo 重复实例必须能删除 exception 并恢复父排除关系。

### 11.2 数据源规则

1. Room 是 App 业务状态主存。
2. Provider 是系统日历互通层，不是唯一事实源。
3. 系统提醒是执行层，不承载业务事实。
4. tag、归档、Undo、检票等不能只依赖 Provider 扩展字段。
5. Provider 扩展字段写入必须 best-effort，不能拖垮主事件写入。

### 11.3 架构调用规则

1. UI/Service/Receiver/Worker 不直接写 Repository/DAO/Database。
2. 后端操作必须走 Center/API。
3. 读操作走 Query API。
4. 写操作走 Operation API。
5. 副作用由 Center/Node 统一编排，不在页面或 Receiver 内临时拼接。
6. 新代码必须通过 `checkArchitectureGuardrails`。

### 11.4 课程规则

1. 课程统一为 `MyEvent(tag=course)`。
2. 课程不再维护独立 Course 运行时主模型。
3. 课程编辑/删除必须复用重复日程 this/future/all 语义。
4. 课程同步不走专用展开分支，统一进入事件同步链。

### 11.5 提醒和胶囊规则

1. 日程写入后通过 `schedule.changed` 驱动提醒和胶囊。
2. 不允许在识别/导入等局部入口重复直接 scheduleReminders。
3. 普通通知、胶囊、MIUI 岛的路由由 Reminder/Notification/Capsule 相关 Center 决策。
4. Receiver 只做壳层路由，不承载复杂业务判断。

## 十四、最终结论

这轮 v47-v57 的核心结论是：重复日程问题不能靠单点补丁彻底解决，因为它横跨数据模型、系统日历、通知、Undo、课程和 UI 展示。最初的 Provider-only 尝试证明了系统日历适合承载日历事实，但不适合作为 App 全部业务状态的唯一来源。

最终保留下来的方向是：

```text
Room 承载 App 业务事实
Provider 承载系统日历互通
系统提醒承载执行层
Center/API/Node 约束调用和副作用
```

这个方向既保留了系统日历重复语义的优势，也给 App 自身的 tag、归档、完成、检票、Undo、课程等业务状态留下了稳定落点。后续开发应继续沿用这条边界，而不是回到 UI/Receiver 直连仓库、Provider-only 或课程独立模型的旧路。
