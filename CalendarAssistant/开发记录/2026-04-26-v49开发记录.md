## 二、同步与颜色标签修复

### 1) 问题现象

- App 内创建的日程同步到系统日历后，再从系统日历同步回来，颜色标签会被覆盖。
- 系统侧事件可能没有自定义 `EVENT_COLOR`，导致本地颜色被系统空值/日历默认色替换。

### 2) 落地结果

- 出站同步时，若本地事件颜色非 0，写入 `Events.EVENT_COLOR`。
- 入站同步颜色优先级调整为：
  1. 系统事件 `EVENT_COLOR != 0`：使用系统颜色
  2. 系统颜色为 0 且本地已有颜色：保留本地颜色
  3. 都没有：回退到日历默认色

**关键文件**

- `app/src/main/java/com/antgskds/calendarassistant/calendar/sync/SystemCalendarSyncManager.kt`

## 三、导出 toast 误报修复

### 1) 问题现象

- 日程数据实际已成功导出，但 toast 提示：
  `导出失败: Cannot access database on the main thread...`

### 2) 根因

- 导出完成后，成功 toast 文案中调用 `viewModel.getTotalEventsCount()`。
- 该方法访问 Room，但执行位置在主线程，触发 Room 主线程访问异常。
- 异常被外层 `catch` 捕获，因此成功导出被误报为失败。

### 3) 落地结果

- 在 `Dispatchers.IO` 中完成导出、文件写入和计数读取。
- 主线程只负责展示已计算好的成功 toast。
- `openOutputStream(uri)` 返回 null 时按真实失败处理。

**关键文件**

- `app/src/main/java/com/antgskds/calendarassistant/ui/page_display/settings/BackupSettingsPage.kt`

## 四、显示范围与归档互斥规则

### 1) 最终范围规则

- 首页：不改语义，只显示选中日期；开启“明日日程”时额外显示明天。
- 全部列表页：显示所有历史未归档日程 + 今天 + 未来 7 天。
- 归档页：显示全部已归档日程，不按时间范围过滤。
- 悬浮窗：
  - 全部：同全部列表页，显示所有历史未归档日程 + 未来 7 天
  - 今日：显示与今天有交集的日程
  - 今日和明日：显示与今天/明天有交集的日程

### 2) 归档互斥规则

- `archivedAt == null`：只允许出现在首页、全部列表页、悬浮窗。
- `archivedAt != null`：只允许出现在归档页。
- 单实例日程严格按归档字段互斥展示。

### 3) 落地结果

- `ScheduleCenter.events` 防御性只保留未归档事件。
- `ScheduleCenter.archivedEvents` 防御性只保留已归档事件。
- `ScheduleDisplayHelper` 展开前过滤已归档事件。
- `MainViewModel`、`LocalHomeQueryApi`、`ArchivesPage` 增加 UI 层防御过滤。
- `AllEventsPage` 移除页面内“未来 7 天”的二次过滤，范围统一由 `MainViewModel` 控制。

## 五、自动归档开关修复

### 1) 问题现象

- 归档页出现大量系统日历同步进来的旧日程。
- 表现类似“过去 7 天之外的日程都会默认进入归档页”。

### 2) 根因

- `MainViewModel` 启动和刷新时无条件调用 `scheduleCenter.autoArchiveExpiredEvents()`。
- `ScheduleCenter` 内部硬编码 `now - 7天` 作为归档阈值。
- 该逻辑没有检查用户是否开启 `autoArchiveEnabled`。

### 3) 最终规则

- 自动归档开关关闭：绝不执行自动归档。
- 自动归档开关开启：对数据库中所有 `endTS < now && archivedAt == null` 的已过期日程执行归档。
- 不按来源区分，本地、识别、CalDAV 同步日程都按同一规则。
- 手动归档不受自动归档开关影响。

### 4) 落地结果

- 自动归档调用前检查 `settings.autoArchiveEnabled`。
- 自动归档阈值改为当前时间，不再硬编码 7 天。
- `StoreRootNode.autoArchiveExpiredEvents` 防御性跳过已归档事件。
- 仍跳过重复母事件，避免一次性归档整个重复系列。

**关键文件**

- `app/src/main/java/com/antgskds/calendarassistant/ui/viewmodel/MainViewModel.kt`
- `app/src/main/java/com/antgskds/calendarassistant/core/center/ScheduleCenter.kt`
- `app/src/main/java/com/antgskds/calendarassistant/store/StoreRootNode.kt`

**注意**

- 本轮修复只阻止后续误归档。
- 已经被旧逻辑写入 `archivedAt` 的数据不会自动恢复，因为无法可靠区分“用户手动归档”和“旧 bug 自动归档”。

## 六、悬浮窗改为具体实例展示

### 1) 问题背景

- 悬浮窗原先直接显示原始 `Event`。
- 对重复日程来说，悬浮窗暴露的是重复母事件，左滑完成、检票、归档等操作存在误伤整个系列的风险。
- 主页/全部列表已使用 `ScheduleDisplayItem`，展示的是具体实例。

### 2) 落地结果

- 悬浮窗日程列表改为 `ScheduleDisplayItem`。
- 重复日程显示具体实例，而不是母事件本体。
- 操作统一通过 `ScheduleDisplayItem.ActionTarget` 路由：
  - `Single(eventId)`：真实单次事件或已物化子事件
  - `RecurringOccurrence(parentId, occurrenceTs)`：重复系列中的虚拟实例
- 右滑归档重复实例时，只归档本次实例。
- 悬浮窗编辑重复实例时默认保存为“仅本次实例”，避免改到母事件。

## 七、悬浮窗 Undo 语义调整

### 1) 问题现象

- 延迟提交模式下，用户左滑完成/检票后悬浮窗 UI 不一定立刻刷新。
- 一分钟内再次左滑 Undo 时，由于数据库尚未变化、虚拟实例 key 可能变化，用户无法明确判断当前状态。

### 2) 最终规则

- 左滑完成/检票/恢复待办：立即写数据库，并刷新 UI。
- 60 秒内 Undo：使用操作前快照恢复旧数据。
- 60 秒内不 Undo：不再做额外写库，当前状态自然生效。
- 新状态操作进入时，确认上一个 pending，只保留最近一次可撤销。
- 同一时间只允许一个完成/检票/Undo 状态操作，快速连滑会被忽略。

### 3) 单次事件 Undo

- 操作前保存完整旧 `Event` 快照。
- Undo 时直接 `updateEvent(oldEvent)`，恢复 `state/endTS/flags/importId/source/color` 等完整字段。

### 4) 重复实例 Undo

- 对虚拟实例 `RecurringOccurrence(parentId, occurrenceTs)`：
  - 操作前保存父事件快照
  - 若操作物化出子事件，记录新 `childId`
  - Undo 时删除新子事件，并恢复父事件快照
- 对已物化子事件：按单次事件快照恢复。
- Pending key 同时覆盖旧 `rec:parentId:occurrenceTs` 和新 `single:childId`，避免物化后无法 Undo。

### 5) UndoManager 语义更新

- `UndoManager` 从“延迟提交管理器”调整为“Undo 窗口管理器”。
- `commitAction` 仅负责确认/清理 pending 状态。
- `rollbackAction` 负责恢复旧数据。
- 新增 `undoNow()`，用于在状态操作互斥锁内等待恢复完成。
