## 二、文本随口记挂起到实况胶囊 / 小米超级岛

### 2.1 问题原因

用户反馈新增的文本随口记挂起到小米超级岛不生效。排查后确认：

- 悬浮窗文本随口记路径只调用 `quickMemoCenter.createTextMemo(clean)` 保存文本。
- 保存结果的 `memoId` 没有继续进入 `CapsuleCommandApi`。
- 既有胶囊链路只支持语音转写 `VOICE_TRANSCRIPTION`，没有文本随口记类型。
- 小米超级岛分发层本身能处理胶囊项，问题主要出在上游没有创建文本随口记胶囊状态。

因此本轮没有复用语音转写胶囊，避免把文本随口记显示成“语音转写”。

### 2.2 新增文本随口记胶囊类型

新增独立胶囊类型：

```kotlin
CapsuleType.TEXT_QUICK_MEMO = 10
```

新增命令接口：

```kotlin
CapsuleCommandApi.showTextQuickMemo(memoId, title, durationMs)
CapsuleCommandApi.clearTextQuickMemo()
```

接入层包括：

- `CapsuleStateManagerCommandApi`
- `CapsuleCenter`
- `CapsuleStateManager`
- `CapsuleMessageComposer`
- `SystemLiveDisplay`
- `NotificationIds.quickMemoText(memoId)`
- `IconUtils`
- Flyme 实况模板图标映射

文本随口记胶囊展示策略：

- `primaryText` 使用随口记正文。
- `secondaryText` 固定为 `随口记`。
- 短标题使用紧凑标题。
- 图标复用 `ic_stat_note`。
- 通知 id 使用独立命名空间 `text-quick-memo`，不与语音转写混用。

### 2.3 悬浮窗保存后触发胶囊

`FloatingScheduleService.handleManualInput(...)` 的文本随口记分支调整为：

```text
createTextMemo(clean)
→ 得到 memoId
→ 若实况胶囊开启，调用 showTextQuickMemo(memoId, clean)
```

这样文本随口记保存后会立即进入统一胶囊链路：

```text
FloatingScheduleService
→ CapsuleCommandApi
→ CapsuleStateManager
→ CapsuleDispatcher
→ 原生实况通知 / Flyme / 小米超级岛
```

### 2.4 点击胶囊打开随口记详情

本轮同时补齐语音和文本随口记胶囊的点击行为。

调整范围：

- `MiuiIslandManager`
- `NativeCapsuleProvider`
- `FlymeCapsuleProvider`

行为：

- 对 `VOICE_TRANSCRIPTION` 和 `TEXT_QUICK_MEMO` 解析胶囊 id 中的 `memoId`。
- content intent 中写入 `MainActivity.EXTRA_OPEN_QUICK_MEMO_ID`。
- 点击原生 / Flyme / 小米胶囊后打开对应随口记详情。

小米超级岛摘要中新增：

```text
TEXT_QUICK_MEMO -> 随口记
```

## 三、天气 2x2 小组件布局边界修复

### 3.1 问题原因

OPPO / ColorOS 用户反馈 2x2 天气小组件中，天气信息会与底部预警条重叠。排查 `widget_weather_2x2.xml` 后确认：

- 位置、图标、温度、副标题直接在 `RelativeLayout` 中堆叠。
- 底部预警条使用 `layout_alignParentBottom="true"`。
- 中间温度 / 天气范围文本没有通过 `layout_above` 与底部预警条建立硬边界。
- OEM 小组件缩放、系统字体或显示大小变化后，文本区域可能侵入预警条区域。

### 3.2 修复方式

本轮只修改：

```text
app/src/main/res/layout/widget_weather_2x2.xml
```

关键调整：

- 保留底部 `widget_weather_alert_container` 固定贴底。
- 新增 `widget_weather_info_container` 作为天气信息容器。
- 信息容器设置：

```xml
android:layout_above="@id/widget_weather_alert_container"
android:layout_below="@id/widget_weather_location"
```

- 温度和天气范围文本放入该容器中。
- 容器 `clipChildren` / `clipToPadding` 开启，避免越界绘制。
- 温度字号由 `42sp` 降到 `38sp`，作为边界修复后的次要视觉优化。

本轮重点不是单纯缩小字号，而是增加严格布局边界，确保预警条上方和天气信息区互不重叠。

## 四、剪贴板识别前台 / 后台链路修复

### 4.1 问题原因

重构后剪贴板链路存在一个关键状态错误：

```text
特权服务监听到剪贴板变化
→ 仍通过普通 ClipboardManager.primaryClip 读取内容
→ Android 10+ 后台读取失败
→ 但前台普通提示又因为 privilegedMonitorActive 被抑制
```

结果是在 Root / Shizuku 设备上可能出现两个通道都不可用：

- 后台服务能看到 logcat 剪贴板事件，但读不到内容。
- 前台打开 App 时的普通确认提示被误跳过。

### 4.2 前台普通识别保底

`ClipboardCodeCenter` 增加明确的前台入口：

```kotlin
triggerForegroundCheck(source)
```

并保留旧 `checkClipboardForPrompt(source)` 作为委托入口。

`MainActivity` 增加触发点：

- `onResume()` 调用 `triggerForegroundCheck("activity_resume")`。
- `onWindowFocusChanged(true)` 调用 `triggerForegroundCheck("window_focus")`。

为避免重复提示，`ClipboardCodeCenter` 增加同来源短防抖：

```text
FOREGROUND_CHECK_DEBOUNCE_MS = 750ms
```

防抖按 source 区分，避免 `activity_resume` 抢掉后续 `window_focus` 检查。

### 4.3 只在后台可靠读取时抑制前台

`ClipboardCodeCenter.setPrivilegedMonitorActive(...)` 增加参数：

```kotlin
setPrivilegedMonitorActive(active: Boolean, canReadClipboard: Boolean = false)
```

前台提示现在只在以下条件同时满足时跳过：

```text
privilegedMonitorActive == true
privilegedClipboardReadAvailable == true
```

这意味着：

- Shizuku Binder 读取可用时，可以由后台自动识别接管。
- Root 只有 logcat 监听但没有可靠剪贴板读取时，不再压掉前台普通识别。
- 特权链路异常时，用户打开 App 仍能看到普通剪贴板确认提示。

### 4.4 Shizuku Binder 读取剪贴板

本轮没有使用以下脆弱方案：

```text
service call clipboard ...
cmd clipboard ...
硬编码 Parcel 解析 shell 输出
```

新增读取方式位于 `PrivilegeManager`：

- `canReadClipboardViaPrivilegedApi()`
- `readClipboardTextPrivileged(context)`

实现要点：

- 仅在 Shizuku 可用且已授权时启用。
- 通过 `SystemServiceHelper.getSystemService("clipboard")` 获取系统剪贴板 Binder。
- 使用 `ShizukuBinderWrapper` 包装 Binder。
- 优先尝试反射隐藏 `android.content.IClipboard$Stub.asInterface(...)`。
- 失败后回退到 Binder transact，并按 Android 版本尝试不同 `getPrimaryClip` 签名。
- 得到 `ClipData` 后用 `coerceToText(context)` 转成文本。

适配的 `getPrimaryClip` 参数形态包括：

```text
getPrimaryClip(String pkg, int userId)
getPrimaryClip(String pkg, String attributionTag, int userId)
getPrimaryClip(String pkg, String attributionTag, int userId, int deviceId)
```

后台服务 `ClipboardCodeMonitorService` 调整为：

```text
检测特权读取能力
→ canReadClipboard=true 时标记后台可接管
→ 剪贴板变化后 readClipboardTextPrivileged(...)
→ ClipboardCodeCenter.autoIngestText(text, source)
```

`autoIngestText(...)` 仍沿用既有去重、处理中、失败短 TTL 逻辑，只在入库成功后标记 handled；失败不会永久吞掉剪贴板内容。

## 八、壁纸磨砂表面与 MIUIX 实验

### 8.1 目标与取舍

本轮在上一轮 UI 零件统一的基础上，继续尝试“导入壁纸后让卡片、弹窗与底部弹层拥有磨砂感”的方案。用户希望可调磨砂强度和遮罩深度，并希望预留 MIUIX 风格的实验后端。

经过评估后，没有直接引入 `miuix-main.zip` 全量依赖，主要原因是其 Kotlin / Compose / AGP 版本与当前工程差异较大，直接合并风险高。最终采用项目内本地实现 `AppGlassSurface`，把复杂度收敛在 App 自己的公共表面层里，普通页面不需要关心具体磨砂算法。

本轮保持的原则：

- 只有导入主界面壁纸后，磨砂实验才有意义。
- 普通卡片 / 弹窗 / 底部弹层都走统一公共壳，不在各页面散落特例。
- 磨砂层只负责背景与遮罩，前景文字、图标、开关、按钮保持清晰。
- 壁纸本身是否模糊仍由原来的 `壁纸模糊` 开关单独控制。

### 8.2 台账与设置

按 `CLAUDE.md` 约束，本轮先登记再开发，新增/调整如下：

- `HelperCatalog`：新增 `壁纸磨砂表面支持`，指向 `ui/components/AppGlassSurface`，作为导入壁纸后卡片、弹窗、底部弹层的公共磨砂表面入口。
- `MySettings`：新增 `appBackgroundGlassEnabled`、`appBackgroundGlassMiuixEnabled`、`appBackgroundGlassBlurRadiusDp`、`appBackgroundGlassOverlayAlphaPercent`，分别控制磨砂启用、MIUIX 实验参数、模糊强度和遮罩深度。
- `ConfigCatalog`：新增四个外观配置项，分别对应磨砂卡片、MIUIX 风格模糊、磨砂模糊强度、磨砂遮罩深度；全部放在 `ConfigDomain.APPEARANCE` 下，并作为开发者可见设置暴露。
- `SettingsViewModel`：新增四个 setter，用于保存和裁剪上述配置。

### 8.3 公共磨砂表面

新增 `ui/components/AppGlassSurface.kt`，实现项目内本地磨砂底座：

- `AppGlassSettings` 汇总当前磨砂状态、壁纸位图、根层尺寸和暗色模式。
- `LocalAppGlassSettings` 和 `AppGlassSettingsProvider(...)` 作为组合本地环境，方便公共组件统一读取。
- `AppGlassSurface(...)` 负责根据开关决定是否启用磨砂：关闭时退回普通实色底；开启时使用壁纸对齐的模糊背景 + 主题色遮罩 + 正常前景内容。
- `AppGlassBlurredBackdrop(...)` 通过 `positionInRoot` 和根层尺寸把壁纸对齐到卡片区域，尽量模拟 backdrop blur，而不是把文字和图标一起模糊掉。
- `glassOverlayColor(...)`、`glassBorderColor(...)` 用于根据主题和参数计算遮罩与边框颜色。

### 8.4 公共组件接入

`ui/components/AppUiComponents.kt` 与 `ui/components/FloatingActionCard.kt` 做了统一接入：

- `AppCard` 在磨砂启用时切到 `AppGlassSurface`，未启用时仍是普通 Material 卡片。
- `AppDialogCard`、`AppDialog`、`AppPickerDialog`、`AppModalBottomSheet`、`AppFloatingContainer` 都接入了统一表面能力。
- `FloatingActionCardSurface` 改为通过 `AppCard` 绘制，避免卡片 / 弹层出现两套外壳逻辑。

### 8.5 根层提供与开发者控制

`MainActivity` 在根组合层提供了 `AppGlassSettingsProvider`，把壁纸位图、根层尺寸和磨砂配置送到整个 UI 树。这样普通页面不用自己传递磨砂参数，也不用各自判断是否导入了壁纸。

`DeveloperPage` 新增“外观实验”区域，提供：

- `壁纸磨砂卡片`
- `MIUIX 风格模糊`
- `磨砂模糊强度`
- `磨砂遮罩深度`

其中 `MIUIX 风格模糊` 目前只是本地近似参数路径，不是直接合并外部 MIUIX 工程。

### 8.6 取舍与验证

本轮没有直接合并 `miuix-main.zip` 全量依赖，主要考虑：

- Kotlin / Compose / AGP 版本跨度大，直接引入风险高。
- 当前先用本地公共磨砂层更稳，也更容易控制卡片、弹窗、底部弹层的统一行为。
- 用户更关心的是“能用、可调、前景清晰”，不是先把外部工程完整搬进来。

验证结果：

- `:app:compileDebugKotlin` 通过。中途出现过 `matchParentSize` 使用位置不对的问题，后改成 `fillMaxSize()` 后重新通过。
- `checkArchitectureGuardrails` 通过。
- `:app:assembleDebug` 通过。
- 仅保留已有的 `MainViewModel.kt:195` 非阻塞 unchecked cast 警告。
