## 二、通知 ID 命名空间整理

### 1) 问题背景

旧链路中多处直接使用 `eventId.hashCode()`、`eventId.toInt()` 或字符串 hash 作为通知 id。普通提醒、实况胶囊、取件初始通知和历史通知残留存在命名空间混用风险：

- 取消普通提醒时可能误取消胶囊通知。
- 删除、归档、完成事件后，旧 notification id 可能没有被完全清理。
- 重复实例使用 `rec:${parentId}:${occurrenceTs}` 作为虚拟 key 时，普通提醒和胶囊 id 需要稳定但互不冲突。

### 2) 设计方案

新增统一通知 id 分配器：

- `app/src/main/java/com/antgskds/calendarassistant/service/notification/NotificationIds.kt`

该对象划分三个命名空间：

```text
standard-reminder  -> 普通提醒通知
live-capsule       -> 实况胶囊/胶囊通知
pickup-initial     -> 取件初始通知
```

同时保留历史 id 清理函数：

- `legacyEventIds(eventId)`：清理旧的 `hashCode/toInt/offset` 方案。
- `legacyKeyIds(key)`：清理重复实例或字符串 key 的旧通知残留。

### 3) 代码落点

- `NotificationCenter.kt`
  - 普通提醒展示改用 `NotificationIds.standardReminder()`。
  - 通知点击 PendingIntent 和动作按钮 PendingIntent 使用同一稳定 notification id 派生 requestCode。

- `CapsuleStateManager.kt`
  - 单次事件胶囊改用 `NotificationIds.liveCapsule(eventId)`。
  - 重复实例胶囊改用 `NotificationIds.liveCapsule(instanceKey)`。
  - 胶囊展示后调用 `cancelLegacyCapsuleNotification()` 清理旧胶囊 id，避免新旧通知并存。

- `NotificationScheduler.kt`
  - 取消提醒时同时清理普通提醒、胶囊、取件初始通知和 legacy id。
  - 取件初始通知改用 `NotificationIds.pickupInitial(eventId)`。

- `ReminderStoreNode.kt`
  - 胶囊开启时只清理普通提醒通知，不误清理活跃胶囊。
  - 事件失效时清理普通提醒、胶囊、取件初始通知和旧 id。
  - 重复实例取消时按 instanceKey 清理普通提醒和胶囊通知。

- `EventReminderReceiver.kt`
  - 普通提醒展示和失效清理统一使用 `NotificationIds.standardReminder()`。
  - 事件无效时补充清理 legacy id。

## 三、截图来源路径保留与隐藏

### 1) 问题背景

为了修复截图/图片识别后编辑弹窗无法显示来源图的问题，识别链路重新传递了 `sourceImagePath`：

```text
HomePage / FloatingScheduleService
  -> RecognitionCenter
  -> DomainEventBus
  -> ContentIngestCenter
  -> ImportCenter
  -> AiEventDraftConverter
  -> Event.description 追加 [img:绝对路径]
```

这让来源图可以保存到事件中，但也带来新问题：`description` 被多个 UI、通知、胶囊和同步入口直接展示，导致截图目录或图片绝对路径显示给用户。

### 2) 设计方案

保留 `[img:...]` 作为内部元数据，不新增数据库字段；所有用户可见展示和外部同步都先剥离该标记。

新增工具：

- `app/src/main/java/com/antgskds/calendarassistant/core/util/SourceImageMarkers.kt`

提供三个函数：

- `extractSourceImagePath(description)`：从备注中读取来源图路径。
- `stripSourceImageMarkers(description)`：从用户可见文本中移除 `[img:...]`。
- `mergeSourceImageMarker(description, sourceImagePath)`：保存时合并干净备注与来源图路径。

### 3) 入库链路调整

- `HomePage.kt`
  - 图片导入识别时传入 `sourceImagePath = imageFile.absolutePath`。

- `FloatingScheduleService.kt`
  - 悬浮窗图片选择/OCR 后传入 `sourceImagePath`。
  - 手动输入识别接口继续保持可选来源图参数。

- `ContentIngestCenter.kt`
  - 识别完成事件入库时不再丢弃 `payload.sourceImagePath`。

- `AiEventDraftConverter.kt`
  - 使用 `mergeSourceImageMarker()` 将来源图路径写入 `description` 内部标记。
  - 避免重复拼接 `[img:...]`。

### 4) 编辑与展示调整

- `AddEventDialog.kt`
  - 初始化时从 `description` 提取来源图路径。
  - 备注输入框显示 `stripSourceImageMarkers()` 后的干净备注。
  - 保存时使用 `mergeSourceImageMarker()` 把来源图路径合并回事件。
  - 仍然根据来源图路径加载缩略图预览。

- `SwipeableEventItem.kt`
  - 首页/全部列表卡片展示备注前剥离来源图标记。

- `FloatingScheduleScreen.kt`
  - 悬浮窗卡片渲染、展开详情、编辑输入框均不显示来源图路径。
  - 悬浮窗编辑保存时保留原来源图标记，避免编辑后丢失截图来源。

## 四、规则展示、通知、胶囊和同步的路径隐藏

### 1) 规则与模板展示

- `RuleMatchingEngine.kt`
  - `resolvePayload()` 和 `extractPayloadText()` 先剥离 `[img:...]`，避免内部图片标记被误识别为规则头。

- `EventPresentationInternals.kt`
  - `sanitize()` 统一剥离来源图标记，普通日程摘要、胶囊展开文案等不再显示路径。

- `RuleDisplayTemplateResolver.kt`
  - `{description}` 占位符使用剥离后的备注。
  - `{payload}` 继续走 `RuleMatchingEngine.extractPayloadText()`，同步受益于路径隐藏。

### 2) 普通通知与提醒

- `EventReminderReceiver.kt`
  - 标准提醒通知正文展示前剥离来源图标记。

- `NotificationScheduler.kt`
  - 取件初始通知 BigText fallback 不再直接展示带 `[img:...]` 的原始备注。

### 3) 胶囊与厂商适配

- `CapsuleStateManager.kt`
  - 生成 `CapsuleUiState.Active.CapsuleItem.description` 时剥离来源图标记。
  - 单个取件、聚合取件、普通日程胶囊描述均避免泄露本地路径。

### 4) 系统日历同步

- `SystemCalendarSyncManager.kt`
  - 写入 Android Calendar Provider 时使用剥离后的 `Events.DESCRIPTION`。
  - App 内仍保留 `[img:...]` 作为内部元数据，系统日历侧不暴露本地截图路径。
