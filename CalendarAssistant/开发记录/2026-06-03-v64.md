# 2026-06-03 开发记录 (v63)：便签编辑器流畅度收尾与 active 窗口改造

## 一、记录范围

本记录补充 `v63` 之后围绕新版便签编辑器继续做的一轮收尾开发，重点处理两类问题：

1. `/t`、`/d` 在同一行连续输入时不能顺滑补全。
2. 单活动编辑行在连续回车、快速切换和大文档中仍然偏脆，体感流畅度不够稳定。

本轮没有调整版本号，当前仍为：

```text
versionName = 2.1.0
versionCode = 62
```

核心修改文件：

- `app/src/main/java/com/antgskds/calendarassistant/ui/components/PlainNoteEditor.kt`

---

## 二、背景

上一轮已经把便签编辑器从“全量多 `EditText`”改成了“只保留当前编辑态”的方向，输入法回车、多行粘贴、软键盘退格这些问题也基本解决。

但继续真机使用后，仍暴露出两个现实问题：

### 2.1 `/t`、`/d` 只能整行触发

旧逻辑要求整行只有 `/t` 或 `/d` 才会触发，导致：

```text
/t + 回车 -> 19:30
19:30 /d + 回车 -> 不能继续补日期
```

这不符合真实使用习惯。用户更自然的期待是：在同一行里连续补时间、补日期，而不是每次都新起一行。

### 2.2 单活动行过于脆弱

虽然“一个 `EditText` + 其他全只读”的方案资源占用最低，但在以下场景中体验仍然偏紧：

- 连续回车拆多行。
- 快速点击附近几行来回改动。
- 行首退格合并后马上继续输入。
- 中等偏长文档中频繁移动焦点。

问题的根源不是单点 API，而是“单活动行 + 频繁全量 `rebuildLines()`”组合在高频交互下过于敏感。

## 三、`/t`、`/d` 改为同行 token 替换

### 3.1 新行为

本轮把 `/t`、`/d` 从“整行命令”改成了“光标前 token 替换”。

现在支持：

```text
/t + 回车
=> 19:30

19:30 /d + 回车
=> 19:30 2026年6月3日

19:30/d + 回车
=> 19:30 2026年6月3日
```

### 3.2 实现方式

回车时不再先看整行，而是：

1. 读取当前光标位置。
2. 检查光标前最后一个 token 是否为 `/t` 或 `/d`。
3. 如果命中，则只替换该 token，不拆新行。
4. 若 token 前后缺少空格，则自动补齐，避免结果粘连。

新增的小函数包括：

- `findInlineShortcutToken()`
- `matchesInlineShortcutToken()`
- `replaceInlineShortcut()`

### 3.3 保留策略

- `/i`、`/f` 暂时仍保持整行命令逻辑。
- 命令仍只在当前 active 普通文本行里触发。
- 没有回到 `TextWatcher` 中做命令识别，仍沿用输入源头拦截方案，避免破坏输入法组合态。

## 四、从单活动行升级为 5 行 active 窗口

### 4.1 为什么不继续坚持只有 1 个编辑态

前面讨论过，单活动行虽然理论上最省资源，但对高频连续操作并不友好。哪怕临时多保留 2 个编辑态，也很容易在“疯狂回车”或快速改相邻多行时出现焦点脆弱的问题。

因此本轮的折中方案是：

```text
以当前焦点行为中心
保留前后各 2 行 + 当前行
= 共 5 行 active window
```

这比“整页全编辑态”更轻，也比“永远只有 1 行编辑态”更稳。

### 4.2 新状态

继续沿用：

- `focusedLineId`

并新增：

- `pendingSelectionOffset`

作用：

- 点击只读行时，先记录目标光标位置。
- 激活为 `EditText` 后，把光标准确恢复到触点附近，而不是永远跳到末尾。

### 4.3 active window 判断

新增：

- `activeWindowRange(centerIndex)`

规则：

- 以当前焦点段落为中心。
- 向前扩 2 行，向后扩 2 行。
- 只让这 5 行保持 `TextLine(EditText)`。
- 其它文本段落回落为 `ReadOnlyTextLine`。
- 图片、文件、分割线继续保持块视图，不进入文本编辑窗口。

## 五、从全量重建转向局部更新 API

### 5.1 上一版的问题

虽然上一版已经引入了 `ReadOnlyTextLine`，但很多操作最终还是会落回：

```kotlin
rebuildLines(...)
```

这意味着：

- 回车拆行时可能整段重建。
- 多行粘贴时可能整段重建。
- 退格合并时可能整段重建。
- 激活/失活附近行时也可能整段重建。

这正是“还有一点点卡”的主要来源。

### 5.2 本轮新增的内部 API

为了把更新粒度压小，本轮补上了第一版局部更新骨架：

- `buildHolder()`
- `replaceLine()`
- `insertLine()`
- `removeLine()`
- `bindLine()`
- `activateLine()`
- `deactivateLine()`
- `refreshTailFrom()`
- `refreshActiveWindow()`

设计目的：

- 单行替换只改一行 View。
- 插入/删除只改当前位置及必要的后续 prefix。
- 焦点变更时，只刷新旧 active window 和新 active window 的差异区域。

### 5.3 `rebuildLines()` 的角色变化

本轮之后，`rebuildLines()` 不再是常规编辑主路径，而主要退回到两类用途：

1. 首次绑定文档。
2. 空文档初始化或强制全量刷新兜底。

普通编辑过程已经尽量切换为局部更新。

## 六、焦点接力与输入主路径改造

### 6.1 点击只读行

现在点击 `ReadOnlyTextLine` 时：

1. 通过 `getOffsetForPosition(x, y)` 获取触点对应字符位置。
2. 调用 `activateLine(index, offset)`。
3. 若该行当前不是 `TextLine`，则只替换这一行。
4. 再刷新 active window 差异区。
5. 最后把焦点落到精确 offset。

### 6.2 回车拆行

普通文本回车时不再优先全量重建，而是：

1. 更新旧行文本为回车前半段。
2. 在数据层插入新 paragraph。
3. 原地 `bindLine(index)` 更新旧行。
4. `insertLine(index + 1, ...)` 插入新行 View。
5. 调整 active window。
6. 让新行 `EditText` 直接接管焦点。

这虽然还不是严格意义上的“完整双 `EditText` 生命周期管理”，但已经从“整段先拆掉再重建”升级成“局部接力”。

### 6.3 多行粘贴

普通行多行粘贴、提交多行文本时，现在会：

- 只移除当前受影响行。
- 插入新的 paragraph 列表。
- 刷新后续必要区域。
- 把焦点放回目标行目标列。

### 6.4 行首退格

当前行与上一行合并时：

- 先更新上一行文本。
- 删除当前行 View。
- 刷新必要的后续行。
- 保持焦点回到合并后上一行的原连接点。

## 七、代码块与工具栏行为

本轮继续保持上一轮已经确定的代码块规则：

- 代码块软键盘回车不拆 paragraph，而是插入 `\n`。
- 物理键盘回车也走同一逻辑。
- 多行粘贴到代码块时保留在同一个代码块中。
- 工具栏把当前段落切到 `CODE` 后，会自动确保后面存在一个普通空行，便于退出代码块继续写正文。

工具栏相关操作这次也改成尽量局部刷新：

- 切换待办。
- 切换段落样式。
- 插入附件。
- 插入分割线。

不再一律走整篇重建。

## 八、真机体验结论

本轮完成后，实际体感是：

- 比“单活动行 + 全量重建”明显稳定。
- 连续回车和相邻多行切换时更自然。
- 大文档里仍然能感觉到轻微的局部刷新成本，但已处于可接受范围。

当前判断是：

- 普通用户很少会做极端连续回车压测。
- 这套 `LinearLayout + 5 行 active window + 局部更新` 方案已经足够支撑现阶段真实使用。
- 如果未来真的出现几千/几万段文档、高频重编辑场景，再考虑迁移到 `RecyclerView + active ViewHolder`。

## 十一、阶段结论

本轮最重要的结论是：便签编辑器的流畅度问题不能只靠“减少 `EditText` 数量”解决，还必须同时控制刷新粒度和焦点接力方式。

最终阶段性方向变为：

```text
块级文档模型
    + 5 行 active window
    + 局部行替换 / 插入 / 删除
    + 输入源头拦截（回车 / 退格 / 粘贴）
    + 全量重建仅作为初始化和兜底
```

这套结构还没有走到 `RecyclerView` 终局，但已经把新版便签编辑器从“能用”推进到了“日常使用足够顺手”的阶段。
