## 二、通知与提醒链路修复

### 2.1 单次日程提醒全量重排补齐

在原有实现中，重复日程实例已经能通过新通知链路按窗口重排，但普通单次日程在部分场景下仍只依赖创建/更新时的局部注册。

本轮补齐了单次日程的 full reconcile 路径：

- `ScheduleNotificationBridge` 新增单次日程批量提交能力。
- `ReminderCenter` 在全量重排时同时提交单次日程和重复实例。
- 设置变化、应用启动、系统广播触发的提醒重排都能覆盖单次日程。
- 对已经 `POSTED` 的提醒保留 `KEEP_POSTED` 行为，避免重复补发。

这样，以下场景可以重新同步到通知注册表：

- 手机/应用重启后的提醒恢复。
- 提前提醒时间、实况胶囊开关变化。
- 同步导入或状态变化后触发的全量重排。
- 已删除或失效单次日程的残留提醒清理。

### 2.2 提醒重排 ANR 修复

测试机出现过多次 `ReminderReconcileReceiver` 相关 ANR。日志显示多个全量提醒重排并发执行，同时对 SharedPreferences 通知注册表反复逐条读写，导致后台广播超过系统超时阈值。

本轮修复：

- `NotificationRegistryStore` 增加批量删除接口 `deleteAll(...)`。
- `SharedPreferencesNotificationRegistryStore` 批量删除时只读写一次 JSON。
- `NotificationApi` / `NotificationCenter` 增加 `cancelAll(...)`。
- `ScheduleNotificationBridge` 的 stale cleanup 和 known-key cleanup 改为批量取消。
- `ReminderCenter` 增加全量重排 single-flight 保护，多个请求会合并为一次或一次后补跑，不再并发执行。

真机验证中可见日志：

```text
ReminderCenter: full reminder reconcile already running; request coalesced
```

说明后续重排请求已被合并，没有再并发卡住注册表。

### 2.3 新通知链路 action 按钮恢复

用户反馈通知按钮“已完成 / 已检票”点击后无反应。排查后确认新普通通知发布器此前没有渲染 `payload.actions`，且日程提醒请求中也没有补齐动作数据。

本轮修复：

- `ScheduleNotificationBridge` 根据日程类型构造 `NotificationAction`。
- 普通单次日程 action payload 使用真实 `eventId`。
- 重复实例 action payload 使用 `rec:<parentId>:<occurrenceTs>`，与 `EventActionReceiver` 的解析逻辑一致。
- `AndroidNormalNotificationPublisher` 渲染 `payload.actions` 为 `NotificationCompat.Action`。
- `EventActionReceiver` 增加接收、解析、执行成功和异常日志，便于以后排查。

真机验证：

- DEBUG 错过补发提醒通过新链路发布。
- 日志显示 `published normal ... actions=1`。
- 手动广播模拟 action 点击后，`EventActionReceiver` 成功把测试日程置为完成。

## 三、过期/导入日程恢复修复

### 3.1 过期判断规则确认

日程是否过期不是持久化字段，而是根据当前时间和 `endTS` 动态判断：

```text
endTS < now => 过期
startTS < now < endTS => 进行中 / 有效
```

用户手动测试时一度误把当天凌晨时间当成下午时间，日志确认测试机系统时间正常，手动编辑的两个日程结束时间仍在当前时间之前，所以保持过期是正确行为。

### 3.2 导入/归档日程编辑恢复

备份导入的日程可能携带：

- `archivedAt != null`
- 非 `STATE_PENDING` 状态
- 旧数据中的异常状态值

如果只改时间，不清理这些失效状态，日程仍会被 UI 和通知链路当成不可提醒对象。

本轮修复：

- `ScheduleCenter.updateSingleFromPatch(...)` 在用户把日程时间改到有效窗口时，归一化为 `STATE_PENDING` 并清空 `archivedAt`。
- `ScheduleCenter.updateEvent(...)` 也接入同样的 active-window edit 归一化，覆盖更多更新入口。
- 时间窗口、状态、归档状态变化后强制走 `ScheduleNotificationBridge.onEventTimeEdited(...)`，取消旧提醒快照并重新注册。
- `ScheduleNotificationBridge` 取消旧提醒时会读取已有 offset，避免只清理固定的 0/30/45/60 分钟。

新增调试动作：

```text
debug:test-imported-restore
```

该动作会创建一条模拟“备份导入后的归档/异常状态日程”，再把它编辑到当前有效窗口。真机日志验证：

```text
active-window-edit ... restored=true ... afterState=0 afterArchived=null
schedule eventId=... action=MISSED_IMMEDIATE
published normal ... actions=1
```

## 四、语音输入与实验室设置整理

### 4.1 语音输入入口移入实验室

语音输入暂不单独做设置页，改为放入实验室页的独立分组：

```text
语音输入
实验功能
开发者
```

其中“语音输入”分组包含：

- 语音输入总开关。
- 悬浮窗长按语音输入开关。
- 语音转写模型导入入口。

同时从偏好设置“操作”板块移走语音转写模型，避免语音相关设置分散。

### 4.2 新增语音输入总开关

新增设置项：

```text
voice.input.enabled
```

并登记到 `ConfigCatalog` 的语音域。关闭后，长按音量+和悬浮窗入口都不能启动语音输入。

相关链路：

- `MySettings.voiceInputEnabled`
- `SettingsTransformApi`
- `LocalSettingsTransformApi`
- `SettingsViewModel.updatePreference(...)`
- `TextAccessibilityService`

### 4.3 悬浮窗长按语音输入防误触

用户反馈呼出悬浮窗后再次长按容易误进入语音。为避免改变旧用户默认行为，本轮新增设置项默认开启：

```text
voice.floating_long_press.enabled
```

效果：

- 只控制“悬浮窗已显示 / 刚呼出悬浮窗后继续长按音量+”进入语音的路径。
- 不影响用户显式选择“长按音量+ = 语音输入”的独立动作。
- 默认 `true`，避免升级后用户原有习惯突然失效。

### 4.4 本地录音测试入口移除

实验室中旧的“本地录音测试”入口和 `LocalRecorderTestActivity` 已移除，避免用户或维护者误以为它仍是正式功能入口。

### 4.5 语音模型卡片样式统一

语音转写模型导入卡片原本复用了旧 `LaboratoryActionCard` 样式，底色、圆角和阴影都与同组开关卡片不同。

本轮统一为：

- `surfaceContainerLow` 底色。
- 16dp 圆角。
- 0dp 阴影。
- 标题字号/字重与同组开关一致。
- 导入按钮保持右侧布局。

## 五、底栏与悬浮窗权限修复

### 5.1 底栏隐藏“今日日程”后无法创建日程

底栏可自定义隐藏/排序，但 `HomeScreen` / `HomePage` 仍使用固定 tab index 判断 `TODAY / ALL / NOTE`，导致隐藏“今日日程”后页面语义错位，底部 `+` 创建行为异常。

本轮改为基于 `HomeEntryKey` 判断当前页面：

- `HomeScreen` 不再把 page key 映射到固定 tab index。
- `HomePage` 使用 `currentPageKey` 分支判断今日、全部、笔记页。
- 底部 `+` 创建入口不再依赖“今日日程”是否显示。

### 5.2 悬浮窗权限通知点击跳转修复

无悬浮窗权限时，原通知点击只打开主界面，不能直达系统悬浮窗权限页。

本轮新增专用通知方法：

```text
NotificationCenter.showFloatingPermissionDeniedNotification(...)
```

点击通知会打开：

```text
Settings.ACTION_MANAGE_OVERLAY_PERMISSION
```

并带当前包名 URI，直接跳转到本应用悬浮窗权限页。
