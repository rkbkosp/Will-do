# 第一部分：项目概述

## 1.1 项目定位

Will do 是一款 Android 智能日程助手，目标是把“识别 -> 入库 -> 同步 -> 提醒 -> 通知/胶囊展示 -> 用户动作回写”做成稳定闭环。

核心能力：

- AI 文本识别、图片 OCR、多模态识别。
- 短信取件码自动解析。
- 普通日程、重复日程、课程、便签统一展示。
- Room 本地主存与 Android Calendar Provider 双向同步。
- 普通通知、实况胶囊、Flyme 胶囊、HyperOS 超级岛适配。
- 悬浮窗、侧边栏、快捷磁贴、长按音量键等多入口操作。
- 归档、完成、检票、Undo、自动归档等业务状态。
- 天气、网速胶囊、每日总结、备份恢复等扩展功能。

## 1.2 当前架构结论

当前架构核心结论：

```text
Room 承载 App 业务事实
Calendar Provider 承载系统日历互通
系统 Alarm/Notification 承载提醒执行层
Center/API/Node 约束调用与副作用
```

Provider-only 方向已经放弃。系统 Calendar Provider 适合承载时间、RRULE、EXDATE、exception 等日历事实，但不适合作为 App 的唯一事实源。`tag`、归档、完成、检票、Undo、便签、课程元数据等都需要 App 自己可控的业务存储。

## 1.4 设计原则

- UI 不直接写数据库，不直接写系统日历。
- Service/Receiver/Worker 不直接绕过 Center 和 Store 主链。
- 重复日程单次变化必须通过 EXDATE + 子事件/exception 表达。
- 课程是特殊日程，使用 `tag = course`，不再维护独立课程主存。
- 识别结果统一进入入库中心，避免每个入口各自 `addEvent`。
- 提醒、胶囊、同步都是副作用，应由中心或 Store 节点统一编排。
- Provider 扩展字段只能 best-effort，不能影响主事件写入成功。

## 4.2 目录设计说明

- `calendar/` 是本地事件模型、Room、系统日历同步的底层能力区。
- `core/center/` 是业务中心层，不应直接写 UI 状态。
- `store/` 是日程数据主链；所有日程持久化和系统同步最终都会到这里。
- `service/` 是 Android 系统入口壳层，不能承载过多业务判断。
- `ui/` 只负责展示和用户交互，后端操作要调用 ViewModel/Center。

## 5.2 `Event`：日程主实体

路径：`app/src/main/java/com/antgskds/calendarassistant/calendar/models/Event.kt`

`Event` 是 Room 表 `events` 的实体，也是 App 日程业务主存的核心模型。

| 字段 | 作用 |
|------|------|
| `id` | Room 自增主键。 |
| `startTS` / `endTS` | 秒级开始/结束时间。 |
| `title` / `location` / `description` | 标题、地点、描述。 |
| `reminder1/2/3Minutes` | 三个提醒时间。 |
| `rrule` | 重复规则；非空且 `parentId == 0` 表示重复母事件。 |
| `exdates` | 被排除的重复实例。 |
| `attendees` | 系统日历参与者映射。 |
| `importId` / `source` | 系统日历映射。 |
| `timeZone` / `flags` | 时区与全天等标志。 |
| `eventType` | 本地事件类型或系统日历映射类型。 |
| `parentId` | 重复例外/脱离系列子事件的父事件 ID。 |
| `color` / `availability` | 颜色与忙闲状态。 |
| `state` | pending/completed/checked_in。 |
| `tag` | 业务标签，如 general、pickup、course、note。 |
| `archivedAt` | 秒级归档时间，`null` 表示未归档。 |

### `Event` 函数组成

| 函数/属性 | 用途 |
|-----------|------|
| `getIsAllDay()` | 根据 `flags` 判断是否全天事件。 |
| `getReminders()` | 将三个 reminder 字段转成 `List<Reminder>`，过滤关闭项。 |
| `getCalDAVEventId()` | 从 `importId` 解析系统 Provider 事件 ID。 |
| `getCalDAVCalendarId()` | 从 `source` 解析系统日历 ID。 |
| `getTimeZoneString()` | 返回合法时区，异常或空值时回退系统默认时区。 |
| `isArchived` | `archivedAt != null`。 |
| `isRecurring` | `rrule.isNotBlank() && parentId == 0L`。 |
| `isException` | `parentId != 0L`。 |

## 7.8 颜色与 EventType 策略

- 每个系统日历对应一个本地 `EventType`。
- 正向同步时，`event.color != 0` 会写入 `EVENT_COLOR`。
- 反向同步颜色优先级：系统事件颜色 > 本地已有颜色 > 系统日历颜色 > 默认色。
- 目标日历选择优先级：事件 `source` > `lastUsedCaldavCalendarId` > 已选日历第一个。

## 7.9 同步注意事项

- `scheduleCalDAVSync()` 当前使用固定周期；设置中的同步间隔并非所有路径都直接使用。
- 反向同步期间必须通过 `SyncLoopGuard` 禁止再次正向推送。
- 系统日历权限缺失时同步链路会安全退出。
- 非本地账号的扩展字段可靠性有限，业务状态不能只放 Provider。

| `RuleRegistry.getPrimaryTransition(ruleId, currentStateId)` | 获取主动作转换。 |
| `RuleMatchingEngine.resolvePayload(event)` | 从 Event 解析规则 payload。 |
| `RuleMatchingEngine.splitFields(payload, size)` | 拆分规则字段。 |
| `RuleActionDefaults.defaultsFor(ruleId)` | 获取内置规则默认动作。 |

当前规则以内置规则和内存缓存为主，`RuleCenter` 中动态增删改启停仍是过渡或空实现。

# 第十九部分：版本演进、限制与维护规则

## 19.1 近期架构演进

| 阶段 | 结论 |
|------|------|
| v47-v51 | 补齐重复日程 this/future/all、EXDATE、Undo、同步止血。 |
| v50-v52 | 尝试 Provider-only，实测后放弃。 |
| v53-v54 | 写操作 OperationApi 化、读操作 QueryApi 化。 |
| v55-v56 | Center/Node 全域收口，拆除旧 AppRepository 语义。 |
| v57 | 课程并入重复日程，移除纯转发 Port 壳层。 |
| v48-v50（重编号后） | 修复重复日程交互、显示范围、悬浮窗、便签、识别失败胶囊等问题。 |

## 19.2 后续开发必须遵守的规则

- 不允许直接修改虚拟重复实例表达单次变化。
- `THIS` 必须物化子事件并给父事件加 EXDATE。
- `THIS_AND_FUTURE` 必须截断旧 RRULE 并创建新系列。
- Room 是 App 业务状态主存，Provider 是系统日历互通层，不是唯一事实源。
- Provider 扩展字段写入必须 best-effort。
- 反向同步期间必须通过 `SyncLoopGuard` 阻止回推。
- UI 不直接写 DAO/Repository；Service/Receiver 不直接拼业务写入流程。
- 开发记录写过程与原因，技术文档写当前结构与长期规则。

## 19.3 当前实现限制与注意事项

- 项目中存在两套提醒相关实现：`NotificationScheduler + AlarmReceiver + NotificationCenter` 与 `ReminderStoreNode + EventReminderReceiver`。
- `ReminderCenter` 已有 `schedule.changed` 消费链，但当前未发现完整 `ScheduleChangedEvent` 生产点。
- `RecognitionCompletedEvent` 带有 `sourceImagePath` 字段，但自动入库链路中该路径传递到 `ImportCenter` 的实现并不完整。
- `RuleCenter` 当前主要读取内置规则，动态增删改启停仍是过渡或空实现。
- 不同 ROM 和账号类型对 Calendar Provider 的 EXDATE、exception、ExtendedProperties 支持不完全一致。
- `connectedDebugAndroidTest` 跑完后，设备上的 app/test 包可能被卸载或状态变化，手测前应重新安装 debug 包。
